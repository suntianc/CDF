# Product

## Register

product

## Users

CDF 面向本地开发者和小团队技术负责人。他们在自己的机器上处理私有代码、规划工作、运行 Agent、调用 MCP 与 Skills，并希望把多步开发任务从聊天入口推进到可观察、可恢复的执行流程。

用户的典型场景是桌面端开发工作台：一边描述目标，一边查看会话、任务审批、工作流节点、模型与工具设置。界面需要支持较高信息密度，同时让风险动作、执行状态和结果归属清楚可见。

## Product Purpose

CDF 是一个离线优先的 Agent 开发工作站。它把自然语言对话、Master Agent 编排、本地工作流、Agent 资产库、MCP/Skill 配置和执行监控放在同一个 Electron 应用中。

成功的产品体验不是炫技，而是让开发者相信每一步自动化都可理解、可控制、可追踪。用户应该能快速发起任务，看到 Agent 正在做什么，理解哪些工具会被调用，并在需要时批准、暂停或调整流程。

## Brand Personality

大胆、前沿、实验，但必须保持开发者工具的清晰边界。CDF 可以有实验室感和技术张力，但不能牺牲可读性、状态辨识和操作信任。

语气应直接、具体、偏工程化：告诉用户系统在做什么、为什么需要批准、下一步会发生什么。避免空泛宣传语，避免把自动化包装成不可解释的黑盒。

## Anti-references

不要做通用 SaaS 营销模板：不要大面积渐变文案、hero-metric 模板、重复图标卡片网格、过度玻璃拟态、无意义的发光装饰或每节都有小号大写 eyebrow 的结构。

不要让暗色界面滑向霓虹玩具感。紫色可以作为品牌和状态强调，但不应变成到处泛滥的装饰色。不要为了“AI 感”牺牲对比度、可扫描性和标准桌面交互习惯。

## Design Principles

1. **控制感优先。** 每个 Agent、工具调用、审批和执行状态都要能被用户定位和理解。
2. **实验感服务于工作流。** 视觉可以大胆，但大胆只用于表达层级、状态和焦点，不用于制造噪音。
3. **本地可信。** 强调离线、本机、私有和可恢复，界面应像工作台而不是云端营销页。
4. **高密度但不浑浊。** 支持多面板、多状态和长会话，同时通过分层、间距和色彩角色保持可读。
5. **标准交互不重造。** 导航、按钮、输入、弹层和审批控件应熟悉可靠，把新意留给工作流可视化和 Agent 执行反馈。

## Accessibility & Inclusion

默认目标为 WCAG AA。正文和关键状态文本需满足 4.5:1 对比度，大号文本至少 3:1；占位文本不能过浅。所有交互控件需要键盘可达、焦点可见、语义清楚。

动效应支持 `prefers-reduced-motion`。常规过渡保持短促，用于表达状态变化、加载、展开和反馈，不做纯装饰性页面编舞。颜色不能作为唯一状态表达，错误、警告、成功和等待状态需要配合文本或图标。
