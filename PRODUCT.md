# Product

## Register

product

## Users

CDF 面向本地开发者、小团队技术负责人和 Agent 工作流构建者。他们在自己的机器上处理私有代码、规划任务、运行 Agent、调用 MCP 与 Skills，并希望把自然语言请求推进成可观察、可审批、可恢复的本地执行流程。

用户的工作环境通常是长时间打开的桌面开发工作台：左侧管理项目和会话，中间发起命令与查看结果，侧边区域处理审批、工具活动、Agent 栈和工作流节点。界面需要支持高信息密度，但不能让用户失去方向感。任何自动化动作都要能被定位、解释和追溯。

## Product Purpose

CDF 是一个离线优先的本地 Agent 指挥舱。它把 Master Agent 对话、本地工作流编排、Agent 资产库、MCP/Skill 配置、工具审批和执行监控放在同一个 Electron 应用中。

产品成功的标志不是“看起来像 AI 产品”，而是让开发者相信系统正在按可理解的方式调度行动。用户应该能快速下达指令，看见 Agent 正在做什么，知道哪些工具会被调用，在关键节点批准或拒绝，并在失败时找到恢复路径。

## Brand Personality

克制、工程化、暗含锋芒。

CDF 的高级感来自操作确定性、系统秩序和局部张力，而不是视觉噱头。它可以有前沿感，但这种前沿感应体现在执行轨迹、状态反馈、工作流编排和本地控制权上。界面像调度系统，不像内容网站；像专业桌面工具，不像紫色 SaaS 仪表盘。

语气应直接、具体、偏工程化。系统应该说明它正在做什么、为什么需要批准、下一步会发生什么。不要把自动化包装成神秘黑盒，也不要用营销语言替代状态解释。

## Anti-references

不要做成普通 AI Chat App、紫色 SaaS Dashboard、VS Code 皮肤、黑绿 Hacker Terminal 或云端营销页。

不要使用大面积渐变文案、hero-metric 模板、重复图标卡片网格、过度玻璃拟态、无意义发光装饰、彩色霓虹边框或每个区域都有小号大写 eyebrow 的结构。紫色不应是泛品牌装饰，它只能在命令、执行、选中、焦点、Agent 编排和关键反馈中出现。

不要为了“AI 感”牺牲对比度、可扫描性、标准桌面交互和本地可信感。界面不应该像在展示能力，而应该像在调度行动。

## Design Principles

1. **指挥舱，不是仪表盘。** CDF 不是展示数据的页面，而是调度 Agent、工具和工作流的操作系统。界面要优先呈现行动、状态和控制点。
2. **系统秩序制造高级感。** 高级感来自清晰的区域关系、稳定的组件语法、可预测的状态变化和一致的执行反馈。
3. **信号色必须有事件含义。** 高亮只在某件事被选中、运行、等待、失败或需要行动时出现。没有事件，就不该发光。
4. **本地可信优先。** 强调离线、本机、私有、可恢复。视觉上要像可靠的本地工具，而不是远程服务广告。
5. **高密度但有航线。** 支持长会话、多面板、多工具调用和工作流图，但用户永远要知道当前焦点、下一步动作和返回路径。
6. **标准控件保持标准。** 按钮、输入、弹层、导航和审批控件应熟悉可靠。独特性留给 Command Surface、Execution Rail、Agent Stack 和 Workflow Canvas。

## Accessibility & Inclusion

默认目标为 WCAG AA。正文和关键状态文本需满足 4.5:1 对比度，大号文本至少 3:1；占位文本不能过浅。所有交互控件需要键盘可达、焦点可见、语义清楚。

动效应支持 `prefers-reduced-motion`。常规过渡保持短促，用于表达状态变化、加载、展开、审批和反馈，不做纯装饰性页面编舞。颜色不能作为唯一状态表达，错误、警告、成功、等待和执行中状态必须配合文字、图标、位置或形态。
