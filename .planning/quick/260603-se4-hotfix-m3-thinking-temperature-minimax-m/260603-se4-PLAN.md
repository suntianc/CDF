**Step 1 — 代码修补**(`src/main/deepagent/llm-adapter.ts`)

定位:line 550-552 的 if-guard 块。
```
if (config.providerType === 'minimax' || config.providerType === 'minimax-overseas') {
  modelConfig.thinking = { type: 'adaptive' };
}
```

修改:在 `modelConfig.thinking = { type: 'adaptive' };` 这一行(line 551)**之后**,新增一行 `delete modelConfig.temperature;`,2 空格缩进对齐。块内最终形态:
```
if (config.providerType === 'minimax' || config.providerType === 'minimax-overseas') {
  modelConfig.thinking = { type: 'adaptive' };
  delete modelConfig.temperature;
}
```

为什么用 `delete` 而不是 `modelConfig.temperature = null` / `undefined`:
- `delete obj.key` 把键从对象上完全移除,`JSON.stringify` 时整个键不存在,符合 Anthropic 协议对 "未设置" 的字面要求
- `obj.key = null` 仍会序列化出 `"temperature": null`,MiniMax 上游对此语义未明
- `obj.key = undefined` 同样会被 JSON.stringify 跳过,但 TS 类型上不干净;且 LineChatAnthropic 构造时 `this.temperature = fields?.temperature` 仍会读到 undefined,可能触发 LangChain 内部默认值覆盖回去,引入不可控行为
- `delete` 是最稳的:直接抹掉键,后续构造时 ChatAnthropic fields 拿到的是 "没有这个键",完全等同于"未传"

严格不要修改:
- 共享 modelConfig 字面量(line 530-535)
- line 536 `if (config.temperature !== undefined) modelConfig.temperature = config.temperature;`(这是公共逻辑,本次不动)
- case 块其他 3 个 provider 的任何路径(deepseek / zhipu / glm-overseas)
- Anthropic 原生 case 块(line 559-582)
- openai / custom / moonshot / qwen / xiaomimimo case 块(line 504-524)
- ollama case 块(line 583-591)
- `patchMiniMaxAssistantRole(model)` 调用(line 554-556)
