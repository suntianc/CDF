## Task 3 — 节点失败原因分类(`error_type` 枚举规范化)

### Files

- `src/shared/types.ts`(`NodeErrorType` 枚举 + `WorkflowNodeRun.error_type` 类型收紧)
- `src/main/workflow/node-executor.ts`(catch 分支分类)
- `src/main/workflow/workflow-runtime.ts`(aborted 归类 + 写入失败行的 error_type)
- `src/renderer/src/components/WorkflowEditor/ExecutionPanel.tsx`(失败节点显示中文标签)
- `src/renderer/src/components/WorkflowEditor/ExecutionHistoryDrawer.tsx`(历史记录的失败展示)

### Action

1. **`src/shared/types.ts`** — 新增并替换:
   ```ts
   export type NodeErrorType =
     | 'timeout'
     | 'tool_error'
     | 'llm_error'
     | 'no_routing'
     | 'aborted'
     | 'unknown';
   ```
   `WorkflowNodeRun.error_type` 类型从 `string` 收紧为 `NodeErrorType`。

2. **`src/main/workflow/node-executor.ts`** — 在 `createAgentNodeExecutor` 闭包内 `catch (err)` 块(第 549-558 行)做错误分类:
   - 提取一个**纯函数** `classifyError(err: unknown, toolCalls: ToolCallRecord[]): NodeErrorType`,放在文件顶部紧邻 `extractJsonCandidate` 之后:
     ```ts
     function classifyError(err: unknown, toolCalls: ToolCallRecord[]): NodeErrorType {
       if (err instanceof AgentTimeoutError) return 'timeout';
       const message = err instanceof Error ? err.message : String(err);
       const lower = message.toLowerCase();
       if (toolCalls.some((t) => t.success === false)) return 'tool_error';
       if (/(network|fetch failed|econn|timeout|auth|401|403|429|rate[-_ ]?limit|api key)/i.test(lower)) {
         return 'llm_error';
       }
       return 'unknown';
     }
     ```
   - 注意:`AgentNotFoundError` 当前直接 `throw` 透传(在 `getAgent` / `getProvider` 时抛),**不会进** catch 块,无需分类
   - catch 块分类后,把新错误包装成 `Error`(保留原 message),但在 `createAgentNodeExecutor` 的返回路径上加一个 `errorType` 字段? — **不可行,因为 graph-builder 是基于 LangGraph StateGraph,异常机制不携带自定义字段**。**替代方案**:在 throw 之前,先在 node-executor 闭包**外层**(即 `return async (state, onLog) => { ... }` 这个 async 函数)内把分类结果暂存到一个**模块级**的 `lastErrorTypeByNode: WeakMap<WorkflowNode, NodeErrorType>` 中(以 node 对象为 key)。这样在 catch 中分类后 `weakMap.set(node, 'tool_error')` 再 throw,workflow-runtime 收到错误时再从 weakMap 取
   - **简化路径**:**直接借助 `err.message` 在 workflow-runtime 处判定**,不引入 WeakMap。改 node-executor:catch 中分类后,设置一个模块级 `let lastClassifiedErrorType: NodeErrorType = 'unknown';`(文件作用域),在 `classifyError` 末尾赋值。catch 块 throw 前赋值。workflow-runtime 拿到错误时,先 `import { lastClassifiedErrorType }` 再读模块变量。**这是最简单的方案**(单线程、同一时刻只跑一个 node-executor 闭包)
   - **审查风险**:此方案在**并发**情况下会错乱(一个 node 在 catch 还没赋值时被另一个 node 覆盖)。CDF 工作流是单线程顺序执行的 LangGraph(没有真正的并发),但 loop / foreach 内的 `await` 仍可能让两个 node 交错。**改用更安全的方案**:用 node 闭包内的局部变量 + 在 throw 时把它打包进错误对象的 `errorType` 属性(自定义 Error 子类 `ClassifiedNodeError extends Error`),workflow-runtime 通过 `instanceof` 或 duck-typing 读出:
     ```ts
     class ClassifiedNodeError extends Error {
       constructor(public readonly errorType: NodeErrorType, message: string) {
         super(message);
         this.name = 'ClassifiedNodeError';
       }
     }
     ```
   - catch 块:
     ```ts
     const errType = classifyError(err, toolCalls);
     const message = err instanceof Error ? err.message : String(err);
     throw new ClassifiedNodeError(errType, `Agent node ${node.id} execution failed: ${message}`);
     ```
   - 但 `AgentNotFoundError` / `AgentTimeoutError` 仍走 `throw err` 透传 — 这两个的 `name` 已是 `'AgentNotFoundError'` / `'AgentTimeoutError'`,workflow-runtime 通过 `err instanceof AgentTimeoutError` 判定为 `'timeout'`,`err instanceof AgentNotFoundError` 归 `'unknown'`(因为没有 `AgentNotFoundError` 枚举值,维持 unknown)

3. **`src/main/workflow/workflow-runtime.ts`**:
   - 顶部 import 增 `AgentTimeoutError, ClassifiedNodeError` from `./node-executor`,`NodeErrorType` from `../../shared/types`
   - 失败路径 INSERT 块(第 323-326 行)新增一个 `errorType` 分类函数:
     ```ts
     function classifyRuntimeError(err: any): NodeErrorType {
       if (err instanceof ClassifiedNodeError) return err.errorType;
       if (err instanceof AgentTimeoutError) return 'timeout';
       return 'unknown';
     }
     ```
   - 第 324 行 INSERT 把硬编码 `'node_error'` 改为 `classifyRuntimeError(err.error)`,并同步改 `nodeErrorEvent.errorType` 字段
   - 第 427 行 `stopWorkflow` 函数:中止时,需要在所有"正在跑"的 node_execution 上写入 `aborted` 状态 + `error_type='aborted'` — 但当前实现不维护"在跑 node"列表(只有 executionId 维度的 aborted 标记)。**简化**:不动 stopWorkflow 函数本体,改为在 `runWorkflow` 主循环里,`activeExecutions.get(executionId)?.aborted` 检测到后,跳出循环时把"已开始但未完成"的 node 标记为 stopped + error_type='aborted':
     - 维护 `startedNodeIds: Set<string>` 在主循环外层,`chunk` 处理时遇到 `node_start` 事件就 add(从 events_snapshot 反推? **更简单**:在 graph-builder 的 `buildWorkflowGraph((node, upstreamNodeIds) => { return async (state) => { ... pushWorkflowEvent(node_start) ... } })` 处记录;但此回调是 buildWorkflowGraph 内部 — **改用 `nodeStartTimes` 已有的 Map,key 是 nodeId**;在 catch(主循环抛出时)遍历 `nodeStartTimes.keys()` 减去 `allNodeOutputs` 的 key 减去 `allErrors.map(e=>e.nodeId)`,得到"已开始未完成"的 nodeId 集合,每个写一行 status='stopped' error_type='aborted' 的 node_run
   - **简化**:**stopped 路径** — 不在 `runWorkflow` 内做"已开始未完成 node 补写"逻辑(避免引入新分支);改为**仅在工作流主 catch(运行时整体失败,不是被 abort)** 设置 status='failed',**abort 路径下,node 既有 completed 行(已成功)或没有行(未跑)**,startedNode 不会被写入(因为中断后,成功的最后一次 stream chunk 在 break 前已经写过 node_run),**已 started 但未完成**的 node **不补写**(用户能通过 execution.status='stopped' 判断整体状态,节点级 stopped 标记不在本 quick 范围内)— 这条是约束收紧,严格遵守"三件具体的事"
   - 替代:在 stopWorkflow 函数里,如果当前 execution 存在,加一个"中断后回调":在 `runWorkflow` 主循环 break 之后、`finalStatus` 赋值之前,扫描 `nodeStartTimes`,把所有 "started 但无对应 success run + 无对应 error run" 的 nodeId 写一行 `status='stopped'`,`error_type='aborted'`。**采用此方案**
