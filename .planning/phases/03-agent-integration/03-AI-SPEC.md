## 1. System Classification

**System Type:** Multi-Agent | CRUD Configuration Management | Hybrid

**Description:**
Phase 3 构建 Agent 资产库、Skills 管理和 MCP 服务器管理的配置界面。这是一个 CRUD 配置管理层（非 AI 执行层），用户定义 Agent 角色及其 LLM/MCP/Skills 资源绑定。真正的 Agent 执行逻辑属于 Phase 4 的工作流引擎范围。

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer Tooling / AI Agent Configuration Management

**User Population:** Developers using an offline desktop IDE-like environment (Electron + React) to define, configure, and manage AI agent roles, Skills scripts, and MCP server connections.

**Stakes Level:** Medium

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|---------------------|-------------------|--------|--------|
| Configuration Binding Integrity | Agent-MCP-Skills junction table has referential integrity; no orphaned FKs | Orphaned junction records after deletion without cascade cleanup | Critical | D-08, D-09, D-10 in 03-CONTEXT |

### Known Failure Modes in This Domain

1. **Skills script encoding corruption** — Non-ASCII characters (中文脚本名、注释) stored incorrectly due to UTF-8 handling gaps between SQLite and IPC transport
2. **Orphaned junction records** — Deleting an Agent or MCP server without cleaning up `agent_mcp_servers` / `agent_skills` junction table rows
3. **Health check timeout without user feedback** — MCP server at `{host}:{port}` doesn't respond within timeout, but UI shows no error state
4. **UUID collision on concurrent creation** — Two Agents created simultaneously with colliding UUIDs (mitigated by `window.crypto.randomUUID()`)

## 2. Framework Decision

**Selected Framework:** LangGraph (Python + TypeScript)

**Version:** v0.2.x (TypeScript SDK)

**Rationale:**
Phase 3 本身是 CRUD 配置管理层，不直接涉及 AI 执行。但 Phase 4 将构建多步骤 Agent 工作流执行引擎，需要状态管理、checkpointing 和审计轨迹。LangGraph 的图结构非常适合建模复杂工作流节点状态转换、并行/串行执行控制。其 TypeScript SDK 与现有 Electron/React/Zustand 架构完全集成，checkpointing 机制为 Phase 4 的工作流持久化提供基础。

Phase 3 阶段不直接使用 LangGraph 运行时，但提前锁定此框架为 Phase 4 做好准备。

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Claude Agent SDK | Claude-only 供应商锁定，Phase 4 需要多模型支持 |
| OpenAI Agents SDK | OpenAI-only 锁定，不适合多 LLM 提供者架构 |
| CrewAI | Python-only，与 Electron TypeScript 栈不兼容 |
| LangChain | 过于泛化，Phase 4 需要精确的状态管理机制 |
| LlamaIndex | RAG 专用，不适合 Agent 工作流建模 |

**Vendor Lock-In Accepted:** Partial — LangGraph 作为工作流建模层不限定 LLM 提供者，但与 LangChain 生态耦合。Phase 4 需注意抽象边界。

## 3. Framework Quick Reference

> Fetched from official docs by `gsd-ai-researcher`. Distilled for this specific use case.

**Phase Context Note:** Phase 3 is CRUD configuration management (Agent/Skill/MCP asset definition stored in SQLite). LangGraph is not invoked in Phase 3 runtime — it is locked in for Phase 4 agent workflow execution. All Phase 3 implementation uses SQLite + IPC + Zustand, not LangGraph.

### Installation
```bash
npm install @langchain/langgraph @langchain/core zod
```
Version: 1.3.2 | Peer deps: `@langchain/core` ^1.1.44, `zod` ^3.25.32 || ^4.2.0 | Node >= 18

### Core Imports
```typescript
import { StateGraph, Annotation, START, END, Command, MemorySaver } from "@langchain/langgraph";
import type { BaseMessage, AIMessage, HumanMessage } from "@langchain/core/messages";
import { z } from "zod";
```

### Entry Point Pattern
```typescript
// 1. Define state schema with Annotation.Root
const StateAnnotation = Annotation.Root({
  messages: Annotation<BaseMessage[]>({
    reducer: (left, right) =>
      Array.isArray(right) ? left.concat(right) : left.concat([right]),
    default: () => [],
  }),
  agentConfig: Annotation<Record<string, unknown>>({ default: () => ({}) }),
});

// 2. Create graph builder
const graphBuilder = new StateGraph(StateAnnotation);

// 3. Add nodes — async functions (state) => Partial<state>
const agentNode = async (state: typeof StateAnnotation.State) => {
  return { messages: [new AIMessage("response")] };
};
graphBuilder.addNode("agent", agentNode);

// 4. Define edges
graphBuilder.addEdge(START, "agent");
graphBuilder.addEdge("agent", END);

// 5. Compile with checkpoint memory
const checkpointer = new MemorySaver();
const graph = graphBuilder.compile({ checkpointer });

// 6. Invoke
const result = await graph.invoke({
  messages: [new HumanMessage("user query")],
  agentConfig: { agentId: "uuid-from-phase-3" },
});
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `Annotation.Root` | Typed state schema factory. Fields can be value channels (LastValue) or reducer channels. | Always — defines the shape of all state flowing through the graph |
| `StateGraph` | Graph builder. Add nodes (async functions), edges (transitions), then compile into a runnable Pregel instance. | Always — the core builder API |
| `START`, `END` | Pseudo-node constants. `__start__` is entry; `__end__` marks terminal nodes. | Always when wiring entry/exit edges |
| `Command` | Return value to steer graph to a specific node. Used for conditional routing and multi-agent handoffs. | When one node decides which node runs next dynamically |
| `MemorySaver` | In-memory checkpoint saver. Persists state after each node, enabling resumption. | Development; swap for `SqliteSaver` or `PostgresSaver` in production |
| `interrupt` / `NodeInterrupt` | Raise `NodeInterrupt` inside a node to pause execution for human approval or external data. | Human-in-the-loop gates, async tool callbacks |
| `task()` / `entrypoint()` | Spawn parallel sub-tasks within a node for fan-out to multiple sub-agents. | When Phase 4 needs parallel agent execution |

### Common Pitfalls

1. **State is immutable within a node — return partial updates, not mutations.** A node receives full state and returns `Partial<State>`. Do not mutate the input.

2. **Reducer type mismatch causes silent state loss.** `Annotation<string>` (no reducer) uses LastValue semantics — last writer wins. For accumulation (e.g., appending messages), provide explicit reducer.

3. **`MemorySaver` is process-local — not shared across Electron IPC boundary.** In Electron, if the renderer creates a `MemorySaver` and main process is separate, state will not persist. Phase 4 must run the compiled graph in a single process or use a shared checkpoint store (SQLite/Redis).

4. **Conditional edges require returning `Command` or a node name string, not raw values.**

5. **Calling `.invoke()` on an uncompiled graph throws `StateGraphInputError`.** Always call `.compile()` before `.invoke()`.

## 4. Implementation Guidance

**Model Configuration:**
- Provider: Configurable via Phase 3 LLM provider store (supports OpenAI/Anthropic/Gemini via `@langchain/core` chat model abstractions)
- Recommended: `claude-sonnet-4-20250514` for agent reasoning, `gpt-4o` as fallback
- Temperature: 0.2–0.4 for tool-use agents (concise, deterministic-ish); 0.7–0.9 for creative nodes
- `max_tokens`: Always set explicitly — e.g., `4096` for tool responses, `8192` for reasoning summaries. Never leave unbounded.

**Core Pattern (Phase 4 — for reference in Phase 3 planning):**
```typescript
// Phase 4 will load Phase 3 config at runtime:
// 1. Fetch agent definition from SQLite via IPC
// 2. Retrieve bound LLM provider, MCP servers, Skills
// 3. Construct StateGraph with those bindings
// 4. Compile and invoke

// Example: Node that reads Phase 3 config and calls MCP tool
const mcpToolNode = async (
  state: typeof AgentStateAnnotation.State
) => {
  const { agentConfig } = state;
  const mcpServers = agentConfig.mcpServers as MCPServerConfig[];

  // Invoke MCP server via HTTP (Phase 3 configured the host/port)
  const response = await fetch(
    `${mcpServers[0].host}:${mcpServers[0].port}/execute`,
    { method: "POST", body: JSON.stringify({ skill: "..." }) }
  );

  return { messages: [new AIMessage(await response.text())] };
};
```

**Tool Use:**
- MCP invocation: HTTP client (built-in `fetch`) to `{host}:{port}` endpoints configured in Phase 3
- Skills execution: `child_process.spawn` for Bash/Python scripts, `eval()` for JS (sandboxed via VM2 or isolated-vm in production)
- Checkpoint persistence: `MemorySaver` in dev; `@langchain/langgraph-checkpoint-sqlite` for production (SQLite aligns with existing `better-sqlite3` usage)

**State Management:**
- LangGraph state flows through the compiled `Pregel` graph instance
- Phase 3 SQLite data (Agent/Skill/MCP configs) is loaded into graph state at invocation time via `agentConfig` field
- Checkpoints stored separately from SQLite — SQLite holds config, checkpoints hold runtime execution state

**Context Window Strategy:**
- `messages` array uses the built-in `messagesStateReducer` reducer to accumulate conversation history
- When messages approach model context limit (~128k for claude-sonnet-4), implement compaction: a dedicated "summarize" node that condenses old messages into a summary, then clears the message list
- Use `Annotation` with a custom reducer that tracks token count and triggers compaction at threshold

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### Structured Outputs with Pydantic

LangGraph uses Zod (via `zod` peer dependency) for schema validation, not Pydantic directly. The pattern is:

```typescript
import { z } from "zod";

// Define output schema as a Zod schema
const AgentResponseSchema = z.object({
  action: z.enum(["continue", "stop", "escalate"]),
  response: z.string(),
  toolCalls: z.array(z.object({
    name: z.string(),
    args: z.record(z.unknown()),
  })).optional(),
});

// Node that returns typed structured output
const agentNode = async (state: typeof StateAnnotation.State) => {
  const response = await llm.invoke([
    new SystemMessage("You are a helpful agent."),
    new HumanMessage(state.messages.at(-1)?.content ?? ""),
  ]);

  // Parse and validate — retry up to 2 times on failure
  let parsed: z.infer<typeof AgentResponseSchema>;
  let retries = 2;
  while (retries > 0) {
    try {
      parsed = JSON.parse(response.content as string);
      AgentResponseSchema.parse(parsed);
      break;
    } catch {
      retries--;
      if (retries === 0) {
        throw new Error(`Failed to parse LLM output after 2 retries: ${response.content}`);
      }
    }
  }

  return {
    messages: [new AIMessage(response)],
    lastAction: parsed.action,
    toolCalls: parsed.toolCalls ?? [],
  };
};
```

**Retry logic:** 2 retries with logging on failure. Surface to user if all retries exhausted. Never silently accept malformed output.

### Async-First Design

LangGraph is async-native. All node functions are `async`. Key patterns:

```typescript
// Correct: async node
const myNode = async (state: State) => {
  const result = await someAsyncOperation();
  return { output: result };
};

// Common mistake: blocking sync code inside async node
// AVOID: using forEach instead of Promise.all for parallel ops
// CORRECT:
const [a, b] = await Promise.all([asyncOpA(), asyncOpB()]);

// Stream vs await: use stream for UX (token-by-token display)
const streamResult = await graph.stream({
  messages: [new HumanMessage("query")],
});
for await (const chunk of streamResult) {
  // yield chunk to UI in real-time
}
```

### Prompt Engineering Discipline

```typescript
// System prompt: loaded from Phase 3 agentConfig.systemPrompt, injected into state
const SYSTEM_PROMPT = (agentConfig: AgentConfig) => `
You are a ${agentConfig.name}.
${agentConfig.description ?? ""}

You have access to the following MCP servers:
${agentConfig.mcpServers.map(s => `- ${s.name} at ${s.host}:${s.port}`).join("\n")}

You have access to the following Skills:
${agentConfig.skills.map(s => `- ${s.name} (${s.scriptType})`).join("\n")}

Always respond with a JSON object matching the schema.
`.trim();

// User prompt: dynamic, from conversation history
const userMessage = state.messages.at(-1)?.content ?? "";

// Never concatenate system + user into one string — keep them separate
// so the LLM can distinguish role boundaries
```

**Token budget:** Always set `max_tokens` explicitly. Budget example for a claude-sonnet-4 call:
- Input context (system + conversation): ~4k tokens budgeted
- Output max: 4,096 tokens
- Total per call: ~8k tokens at ~$0.003/1k tokens = ~$0.024/call

### Context Window Management

For multi-agent conversations (Phase 4):

```typescript
// Use messagesStateReducer to accumulate messages
const StateAnnotation = Annotation.Root({
  messages: Annotation<BaseMessage[]>({
    reducer: (left, right) =>
      Array.isArray(right) ? left.concat(right) : left.concat([right]),
    default: () => [],
  }),
  messageCount: Annotation<number>({
    reducer: (left, right) => left + 1,
    default: () => 0,
  }),
});

// In a dedicated compaction node:
const compactNode = async (state: typeof StateAnnotation.State) => {
  if (state.messageCount < 50) return {}; // no-op yet

  const toSummarize = state.messages.slice(0, -10);
  const summary = await llm.invoke([
    new SystemMessage("Summarize this conversation concisely."),
    new HumanMessage(JSON.stringify(toSummarize)),
  ]);

  return {
    messages: [
      new SystemMessage(`Summary: ${summary.content}`),
      ...state.messages.slice(-10),
    ],
    messageCount: 0,
  };
};
```

### Cost and Latency Budget

**Cost per call (claude-sonnet-4-20250514 as baseline):**
- Input: ~$0.003/1k tokens
- Output: ~$0.015/1k tokens
- A typical agent turn (4k in, 1k out): ~$0.027

**Caching strategy:**
- Exact-match: cache agent config by UUID — skip graph reconstruction if config unchanged
- Semantic: if systemPrompt hash matches, reuse compiled graph instance
- Sub-task routing: use cheaper model (`claude-3-haiku`) for classification/routing decisions

**Latency budget:**
- Target: < 2s for simple agent turn (no tools)
- With MCP tool call: add network latency (Phase 3 health checks give baseline RTT)
- Stream first tokens: target < 500ms time-to-first-token for UX responsiveness

## 5. Evaluation Strategy

### Phase Context

Phase 3 is a **CRUD configuration management system** — not an AI execution engine. It stores Agent/Skill/MCP configurations in SQLite for consumption by Phase 4's workflow engine. All evaluation dimensions focus on **data integrity, referential correctness, and configuration completeness** — no LLM generation is evaluated in this phase.

### Dimensions

| Dimension | Rubric | Measurement Approach | Priority |
|-----------|--------|---------------------|----------|
| **Referential Integrity** | PASS: Junction tables (`agent_mcp_servers`, `agent_skills`) contain only valid FK references; deleting an Agent/Skill/MCP cascades cleanup to junction records.<br>FAIL: Orphaned junction records remain after parent deletion; INSERT fails due to FK constraint violation. | Code: SQLite FK enforcement enabled; integration tests verify cascade behavior on delete. | Critical |
| **Skills Script Content Fidelity** | PASS: A Skills script saved with UTF-8 Chinese characters (`中文注释`, `脚本内容`) is retrieved byte-for-byte identical.<br>FAIL: Non-ASCII characters corrupted (mojibake), truncated, or silently replaced with replacement characters. | Code: Round-trip test — write known UTF-8 string, read back, assert byte-equality. | High |
| **MCP Health Check Accuracy** | PASS: Health check returns `{ status: "healthy", rtt_ms: <value> }` for reachable server; returns `{ status: "unreachable", error: "..." }` for unreachable server within 5s timeout.<br>FAIL: False positive (server unreachable but returns healthy); false negative (server reachable but times out silently without user-visible error). | Code: Mock HTTP server responds 200; mock HTTP server hangs indefinitely; verify correct status and error messages. | High |
| **Resource Deletion Safety** | PASS: Deleting an Agent/Skill/MCP with bound resources triggers confirmation dialog listing affected bindings; cascade is applied atomically with user awareness.<br>FAIL: Silent cascade delete with no confirmation; deletion fails leaving partial state; error message is unclear. | Code: Integration tests verify confirmation prompt behavior and atomic transaction rollback on failure. | High |
| **Agent Configuration Completeness** | PASS: Creating an Agent with required fields (`name`, `provider_id`) persists correctly; timestamps (`created_at`, `updated_at`) are set automatically; UUID is unique.<br>FAIL: Missing required fields allowed through; timestamps are null or incorrect timezone; duplicate UUID generated. | Code: Zod schema validation on input; DB constraint tests for required fields and uniqueness. | Medium |
| **UI-DB State Consistency** | PASS: After any CRUD operation, the Zustand store reflects the database state within 1 second; no stale data displayed after refresh.<br>FAIL: Store shows deleted record; store does not reflect newly created record; store and DB diverge after concurrent operations. | Code: Integration tests simulate full CRUD cycle and verify store snapshot matches DB query. | High |

## 6. Guardrails

### Phase 3 Context

Phase 3 is a configuration-only system with no AI model invocation. Catastrophic failures in this phase are **data corruption** (not safety violations). Guardrails focus on preventing silent data loss and ensuring downstream Phase 4 execution has valid configuration.

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **Zod Schema Validation on IPC Payloads** | Any IPC handler receives payload that fails Zod schema validation | Block: Return `{ error: "validation_failed", details: zodError }` to renderer; log to `electron-log`; do not persist to SQLite |
| **Required Field Enforcement at DB Layer** | INSERT into `agents` table with null `name` or `provider_id` | Block: SQLite throws `CHECK constraint failed`; catch and return user-facing error "Agent name is required" |
| **Foreign Key Constraint Enforcement** | INSERT into `agent_mcp_servers` with non-existent `agent_id` or `mcp_server_id` | Block: SQLite throws `FOREIGN KEY constraint failed`; catch and return "Referenced Agent/MCP server does not exist" |
| **Health Check Timeout Enforcement** | MCP health check exceeds 5s timeout | Flag: Return `{ status: "timeout", rtt_ms: 5000 }` with user-visible warning; do not block UI |

### Offline (Flywheel)

| Metric | Measurement Approach | Sampling Strategy | Action on Degradation |
|--------|---------------------|------------------|----------------------|
| **Junction Table Orphan Rate** | Weekly batch job: query junction tables for orphaned FKs (LEFT JOIN where parent is null) | 100% of junction tables scanned weekly | Alert: PagerDuty to QA Engineer if any orphans found; fix required within 24h |
| **Skills Script Encoding Error Rate** | Weekly batch: round-trip test all Skills content, flag any non-UTF-8 content | 10% random sample of Skills content tested weekly | Alert: DevOps if any encoding errors detected |
| **Health Check False Positive Rate** | Weekly: compare health check results against actual connectivity (ping test) | All MCP servers tested weekly | Alert: DevOps if false positive rate > 5% |
| **Configuration Completeness Score** | Weekly: count Agents with null `provider_id` / empty `name` as % of total | 100% of agents scanned weekly | Alert: Senior Dev if > 0% incomplete configs |

### Key Metrics to Track

| Metric | What It Measures | Measurement |
|--------|-----------------|-------------|
| **IPC Handler Latency (p50, p95, p99)** | Time from renderer `invoke()` to DB operation completion | OpenTelemetry span duration on `db:*` handlers |
| **CRUD Operation Success Rate** | % of create/read/update/delete operations that succeed | Count of span.status = OK vs. span.status = ERROR |
| **Health Check Latency** | RTT for MCP server health checks | Span duration on `db:checkMcpHealth` |

| **Config Load Time (Phase 4 prep)** | Time to load all Agent configs from SQLite at startup | OpenTelemetry trace from `app.on("ready")` to first workflow invoke |

### Alert Thresholds

| Metric | Warning Threshold | Critical Threshold | Action |
|--------|-----------------|-------------------|--------|
| IPC p95 latency | > 500ms | > 2000ms | Log for investigation; page if sustained > 5min |
| CRUD success rate | < 99% | < 95% | Alert QA; create bug ticket |
| Health check RTT | > 3000ms | > 5000ms (timeout) | Flag MCP server in UI; log for investigation |
| Store sync lag | > 2000ms | > 5000ms | Alert; investigate IPC communication breakdown |

### Smart Sampling Strategy

Since Phase 3 is configuration-only (low user-facing risk per operation), sampling for human review focuses on **anomaly detection** rather than percentage-based sampling:

1. **Sample all failed operations** — any CRUD operation that returns an error is automatically queued for human review within 24h
2. **Sample operations on junction tables** — 20% of Agent-MCP and Agent-Skills bind/unbind operations reviewed weekly
