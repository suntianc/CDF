# Wave 1: Zustand Stores — Agent, Skills, MCP

## Objective

Create three Zustand stores that connect the frontend UI to the Phase 3 backend IPC methods. Each store follows the same pattern as `src/renderer/src/stores/llmStore.ts`: loading/error state, CRUD methods via `window.electronAPI.db.*`, and auto-refresh after mutations.

<action>
Create `src/renderer/src/stores/agentStore.ts` with a Zustand store named `useAgentStore`.

The store must manage `Agent[]` (list state) and provide CRUD via IPC:

```
interface AgentState {
  agents: Agent[];
  isLoading: boolean;
  error: string | null;
  fetchAgents: () => Promise<void>;
  saveAgent: (agent: {
    id: string;
    name: string;
    description?: string;
    provider_id?: string;
    system_prompt?: string;
    config?: Record<string, unknown>;
    mcpServerIds?: string[];
    skillIds?: string[];
  }) => Promise<void>;
  deleteAgent: (id: string) => Promise<void>;
}
```

Key implementation details:
- Uses `window.electronAPI.db.getAgents()` — returns agents WITHOUT resolved `mcpServerIds` and `skillIds` (backend currently returns flat agent rows)
- `saveAgent` calls `window.electronAPI.db.saveAgent()` with the agent payload
- `deleteAgent` calls `window.electronAPI.db.deleteAgent(id)`
- All mutations call `fetchAgents()` after success to refresh state (follow llmStore pattern)
- Error handling: set `error` state, re-throw so callers can show toasts
- Export as named export: `export const useAgentStore = create<AgentState>(...)`
</action>
