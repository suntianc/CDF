# Phase 03.2: deepagents 官方文档系统性复核 - Research

**Researched:** 2026-05-27
**Domain:** deepagents.js SDK API 兼容性与最佳实践审查
**Confidence:** HIGH

## Summary

本研究对项目中 deepagents.js (v1.10.2) 的集成代码进行了全面审计，对照官方文档逐模块比对。项目当前使用 `createDeepAgent` 创建运行时、`FilesystemBackend` 管理文件系统、`MultiServerMCPClient` 集成 MCP 工具、自定义 middleware 处理错误恢复和重试、以及 `registerHarnessProfile` 配置模型行为。

研究发现若干 API 使用与官方文档不一致之处，包括：(1) 子代理配置中 `responseFormat` 使用了 Zod schema 但文档推荐的方式可能有版本差异；(2) MCP 传输类型使用 `sse` 但官方文档已推荐 `http`；(3) `FilesystemBackend` 的 `virtualMode` 使用正确但缺少 `CompositeBackend` 包装可能导致内部数据写入项目目录；(4) 流式处理使用了 `streamEvents` v3 API 而非官方推荐的 `stream()` API。

**Primary recommendation:** 按用户指定的 5 个模块顺序逐步复核，每个模块完成后出具报告供用户确认。

## Module 2: 环境操作与上下文管理

### 官方文档要点

**Backend 体系** [CITED: docs.langchain.com/oss/javascript/deepagents/backends]:
- `StateBackend` — 默认，线程作用域，文件存在 langgraph state 中
- `FilesystemBackend` — 真实文件系统，需要 `virtualMode: true` 启用安全沙箱
- `CompositeBackend` — 路由不同路径到不同 backend
- `StoreBackend` — 跨线程持久化存储

**Memory** [CITED: docs.langchain.com/oss/javascript/deepagents/memory]:
- `memory` 参数接收 AGENTS.md 文件路径数组
- 文件在 agent 启动时加载到 system prompt
- 支持 agent-scoped、user-scoped、organization-scoped 三种作用域

**Context Management**:
- 官方内置 `SummarizationMiddleware` 自动在接近 token 限制时总结对话历史
- 大型工具输入/结果自动 offload 到虚拟文件系统

## Module 4: 人机协同

### 官方文档要点 [CITED: docs.langchain.com/oss/javascript/deepagents/human-in-the-loop]

**interruptOn 配置：**
- `true` — 默认中断，允许 approve/edit/reject/respond
- `false` — 不中断
- `{ allowedDecisions: [...] }` — 自定义允许的操作

**决策类型：**
- `approve` — 使用原始参数执行
- `edit` — 修改参数后执行
- `reject` — 跳过执行
- `respond` — 返回用户消息作为工具结果

**关键要求：**
- 必须有 checkpointer
- resume 时必须使用相同的 thread_id
- 多个工具调用的中断会批量处理

## Common Pitfalls

### Pitfall 1: FilesystemBackend 未用 CompositeBackend 包装
**What goes wrong:** Agent 内部数据（offloaded tool results、conversation history）写入项目目录，污染项目文件
**Why it happens:** 直接使用 `new FilesystemBackend({ rootDir: project.path })`
**How to avoid:** 使用 `CompositeBackend` 包装，将项目文件路由到 `/workspace/`，内部数据路由到 `StateBackend`
**Warning signs:** 项目目录中出现 `/large_tool_results/`、`/conversation_history/` 等非项目文件

### Pitfall 2: SubAgent tools 完全覆盖继承
**What goes wrong:** 子代理指定 `tools` 后失去主 agent 的内置工具（文件操作、bash 等）
**Why it happens:** 官方文档明确指出 `tools` 字段指定后完全覆盖继承
**How to avoid:** 在子代理 tools 中显式包含需要的内置工具，或不指定 tools 让其继承
**Warning signs:** 子代理无法执行文件操作或 bash 命令

### Pitfall 3: MCP SSE 传输类型过时
**What goes wrong:** 使用 `transport: 'sse'` 可能与新版 MCP 服务器不兼容
**Why it happens:** MCP 规范已将 SSE 替换为 Streamable HTTP
**How to avoid:** 更新为 `transport: 'http'`（如果 MCP 服务器支持）
**Warning signs:** MCP 连接失败或超时

### Pitfall 4: streamEvents v3 vs stream() API
**What goes wrong:** 使用旧版 `streamEvents` API 可能错过新的 streaming 功能
**Why it happens:** 历史代码使用 v3 API
**How to avoid:** 评估迁移到 `agent.stream()` + `streamMode` + `subgraphs: true` 的可行性
**Warning signs:** 无法获取子代理的独立流式事件

## Open Questions

1. **CompositeBackend 迁移风险**
   - What we know: 官方推荐使用 CompositeBackend 包装 FilesystemBackend
   - What's unclear: 迁移后对现有 checkpoint 数据的兼容性
   - Recommendation: 在复核阶段评估影响，如无破坏性则直接迁移

2. **streamEvents vs stream() API 迁移**
   - What we know: 项目使用 streamEvents v3，官方推荐 stream() + streamMode
   - What's unclear: 两者在子代理事件获取上的行为差异
   - Recommendation: 暂不迁移，在复核中对比两者行为差异

3. **子代理 tools 覆盖问题**
   - What we know: 当前子代理只传入 MCP 工具，失去内置工具
   - What's unclear: 子代理是否需要文件操作等内置工具
   - Recommendation: 根据实际使用场景决定是否补充内置工具

4. **respond 决策类型**
   - What we know: 官方文档定义了 4 种决策类型，项目只支持 3 种
   - What's unclear: 是否有场景需要 respond 类型
   - Recommendation: 在人机协同模块复核时评估
