按以下步骤逐一检查并修复（per D-03, D-04, D-05）：

**检查项 1: 子代理 tools 覆盖问题**（per 研究问题 3.2 — critical）
当前 runtime.ts 第 527 行 `tools: [...subMcpRuntime.tools]` 只包含 MCP 工具。官方文档明确指出：指定 `tools` 后会**完全覆盖**继承的工具。子代理将无法使用 delete_file、bash、fetch、search 等内置工具。

修复方案：
1. 将主 Agent 的 `builtInTools` 提取为独立变量（如果尚未提取）
2. 在子代理配置中，tools 应包含子代理的 MCP 工具 + 内置工具：
   `tools: [...subMcpRuntime.tools, ...builtInTools]`
3. 注意：builtInTools 需要在子代理配置循环之前定义，或者为每个子代理独立创建

**检查项 2: 子代理 skills 配置**（per 研究问题 3.7）
当前子代理配置中没有设置 `skills` 参数。官方文档指出：自定义 subagent 不继承主 agent 的 skills，需要显式指定。

检查当前代码第 511 行已经为子代理解析了 `subSkillsSources`，但没有传入 subagent 配置。修复：
在 subagents.push 配置中添加 `skills: subSkillsSources.length > 0 ? subSkillsSources : undefined`

**检查项 3: systemPrompt 中的 JSON Schema 冗余**（per 研究问题 3.5）
当前第 526 行在 systemPrompt 中手动拼接了 JSON Schema 指令，但第 530 行已设置 `responseFormat: DELEGATED_TASK_RESULT_SCHEMA`。当 responseFormat 设置后，deepagents 会自动处理结构化输出格式提示。

修复方案：
1. 从 systemPrompt 中移除手动拼接的 JSON Schema 指令
2. 保留 agentRow.system_prompt 原始内容
3. 确认 responseFormat 自动生成的提示是否足够

**检查项 4: responseFormat Zod schema 兼容性**
确认 DELEGATED_TASK_RESULT_SCHEMA（shared/types.ts 第 4-12 行）的 Zod schema 格式与 deepagents 的 `SupportedResponseFormat` 类型兼容。检查 deepagents 版本要求（>= 1.8.4，当前 1.10.2 满足）。

**检查项 5: task 工具输入格式**
确认 llm.ts 中 `buildTaskPackage` 函数（第 21-30 行）生成的 `{ name, task: JSON.stringify({ name, goal }) }` 格式是否与 deepagents 的 task 工具输入约定一致。
