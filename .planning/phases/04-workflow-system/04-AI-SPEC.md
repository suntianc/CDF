## 1. System Classification

**System Type:** Multi-Agent Workflow

**Description:**
Workflow System 是一个基于图的 Agent 编排引擎。用户通过 ReactFlow 可视化编辑器创建工作流，定义 Agent 节点和控制流（分支、循环、并行），系统使用 LangGraph.js StateGraph 作为执行引擎运行工作流。每个 Agent 节点引用资产库中的 Agent 定义，继承其 LLM/MCP/Skills 配置。工作流可从编辑器直接运行或由 Chat 中的 Master Agent 触发，执行状态实时推送到前端，完整执行历史持久化到 SQLite。

**Critical Failure Modes:**
1. 工作流死循环 — 循环路由无最大迭代次数保护导致无限执行
2. 状态丢失 — checkpoint 写入失败导致执行中断后无法恢复
3. Agent 节点静默失败 — 节点执行异常未被捕获和上报，导致下游节点基于错误状态继续执行
4. 并行执行竞态 — fan-out 节点并发写入共享 state 导致数据不一致
5. 跨会话状态污染 — workflow runtime 与 chat runtime 共享 checkpoint namespace

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** developer tooling

**User Population:** 个人开发者，使用桌面 Agent 开发工作站进行自动化开发流程编排

**Stakes Level:** Medium

**Output Consequence:** 工作流执行结果直接影响开发效率；错误的工作流可能执行破坏性操作（如错误的代码修改、数据库操作），但所有操作均在本地环境，可回滚

### What Domain Experts Evaluate Against

**Dimension: 图-执行一致性 (Graph-Execution Fidelity)**
Good: 编辑器中看到的节点连线与实际执行路径完全一致；条件分支走的哪条边、循环迭代了几次，都能在执行回放中精确对应到图上的边
Bad: 执行走了编辑器中未显示的路径，或某条边在编辑器中标记为"已执行"但实际未被触发；用户无法从执行日志还原出真实的图遍历顺序
Stakes: Critical
Source: 工作流编辑器核心价值 — 开发者依赖可视化图来理解和调试自动化流程，图与执行不一致会让调试完全失效（参考 n8n、Zapier 等成熟工作流产品的用户反馈）

**Dimension: 错误可定位性 (Error Observability)**
Good: 节点失败时，错误信息包含：失败节点 ID 和名称、具体错误原因（LLM 调用超时 / MCP 工具返回异常 / 状态校验失败）、失败时的 state 快照、重试次数和结果。用户在 30 秒内能定位根因
Bad: 错误信息只有"workflow execution failed"或一个通用异常堆栈；用户需要逐个展开节点才能找到哪个节点出了问题；LLM 返回格式错误被静默吞掉，下游节点基于空输出继续执行
Stakes: Critical
Source: 开发者工作流调试经验 — 生产环境中工作流失败的首要需求是快速定位问题节点和原因，而非重新执行整个流程（参考 LangGraph 社区常见 issue 中关于 error propagation 的讨论）

**Dimension: 循环安全 (Loop Safety)**
Good: 所有循环边都强制配置最大迭代次数（硬限制），达到上限后工作流停止并报告"循环保护触发"，包含已迭代次数和最后一步输出
Bad: 循环边没有迭代上限，Agent 输出始终满足循环条件时工作流永远不会结束；或迭代上限存在但仅在 UI 层配置，执行引擎未强制校验
Stakes: Critical
Source: LangGraph.js 已知陷阱 — 框架本身不强制循环终止，必须在 state 中实现计数器并在 router 中检查（参考 LangGraph.js 官方文档 Common Pitfalls）

**Dimension: 断点恢复 (Checkpoint Recovery)**
Good: 工作流执行中断（应用崩溃、用户手动停止）后，可从最近的 checkpoint 恢复执行，已成功的节点不重复执行，共享 state 完整保留
Bad: 中断后只能从头重新执行；或 checkpoint 写入失败但未报错，恢复时发现 state 丢失导致下游节点基于空状态执行
Stakes: High
Source: LangGraph.js checkpointing 机制 — SqliteSaver 在每个 superstep 保存 state 快照，但 thread_id 命名空间必须与 chat runtime 隔离（D-21），否则会读写错误的 checkpoint（参考 LangGraph.js persistence 文档）

**Dimension: 并行执行数据一致性 (Parallel Execution Consistency)**
Good: fan-out 并行节点各自写入 nodeOutputs 的独立 key，fan-in 合并后所有并行结果完整存在；无数据覆盖或丢失
Bad: 并行节点写入同一 state 字段时后写覆盖先写，导致部分并行结果丢失；或 fan-in 节点在并行分支未全部完成时就开始执行
Stakes: High
Source: LangGraph.js fan-out/fan-in 模式 — 必须为并行写入的字段定义 ReducedValue reducer，否则默认行为是覆盖（参考 LangGraph.js map-reduce 文档）

### Known Failure Modes in This Domain

1. **视觉-执行漂移 (Visual-Execution Drift)**: 用户在编辑器中修改了节点配置或连线后直接运行，但执行引擎使用的是上一次保存的旧版本。ReactFlow 编辑器的本地 state 与 SQLite 持久化不同步，导致"我改了但没生效"的困惑。这是可视化工作流工具最常见的用户投诉来源。

2. **Agent 静默降级 (Silent Agent Degradation)**: Agent 节点引用的 LLM 提供者不可用（API key 过期、本地模型未启动、MCP 服务器断连），但节点执行不报错而是返回空结果或默认值，下游节点基于错误的"成功"状态继续执行。在 Agent 编排场景中，一个节点的静默失败会级联污染整条执行链。

3. **状态命名空间污染 (State Namespace Collision)**: workflow runtime 与 chat runtime 共享同一个 SQLite checkpointer 或使用重叠的 thread_id 前缀，导致工作流执行读取到聊天会话的 checkpoint，或反之。这在当前项目中是高风险问题，因为 Phase 2 已有 chat runtime 使用 SqliteSaver（D-16a 明确要求独立 runtime）。

4. **上下文窗口溢出 (Context Window Overflow)**: 长链式工作流（>8 个节点）中，上游节点的输出累积传递给下游节点，超出 Agent 的 LLM 上下文窗口限制。表现为后续节点的 LLM 调用截断或报错，但错误信息不明确指出是上下文溢出。

5. **循环边界条件遗漏 (Loop Edge Case)**: 循环路由的终止条件依赖 Agent 输出的结构化判断（如 `{next: "stop"}`），但 Agent 输出格式不稳定（偶尔返回非预期结构），导致循环条件判断失败而继续迭代。需要在 router 层增加格式校验和 fallback 逻辑。

## 2. Framework Decision

**Selected Framework:** LangGraph.js (@langchain/langgraph)

**Version:** latest (to be pinned at implementation)

**Rationale:**
LangGraph.js 是唯一满足所有硬约束的框架：完整 TypeScript 支持、通过 LangChain 生态实现模型无关、原生 StateGraph 支持分支/循环/并行执行、内置 checkpointing。项目已安装 `@langchain/core`、`@langchain/anthropic`、`@langchain/openai`、`@langchain/ollama`、`@langchain/langgraph-checkpoint-sqlite` 等 LangChain 生态包，LangGraph.js 可零摩擦集成。它是为状态化图工作流编排这一精确用例而构建的。

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain (chains only) | 缺乏原生图 checkpointing 和并行 fan-out/fan-in 能力，无法满足 D-15 并行执行需求 |
| CrewAI | Python-only，项目为 TypeScript 技术栈 |
| OpenAI Agents SDK | OpenAI vendor lock-in，违反模型无关约束 |
| Claude Agent SDK | Anthropic vendor lock-in，违反模型无关约束 |

**Vendor Lock-In Accepted:** No — LangGraph.js 开源且模型无关

### Entry Point Pattern — StateGraph as Workflow Execution Engine

This is the **core pattern** for converting a ReactFlow graph into a LangGraph StateGraph. Each ReactFlow node becomes a LangGraph node; each ReactFlow edge becomes a LangGraph edge.

```typescript
import {
  StateGraph, StateSchema, ReducedValue, MessagesValue,
  START, END, Send, MemorySaver
} from "@langchain/langgraph";
import type { GraphNode } from "@langchain/langgraph";
import * as z from "zod";

// 1. Define workflow state schema (D-16b)
const WorkflowState = new StateSchema({
  // User-provided inputs
  inputs: z.record(z.unknown()),
  // Per-node outputs: { [nodeId]: nodeOutput }
  nodeOutputs: z.record(z.unknown()).default(() => ({})),
  // Routing metadata for conditional edges
  routing: z.record(z.string()).default(() => ({})),
  // Artifacts produced during execution
  artifacts: z.array(z.unknown()).default(() => []),
  // Error tracking
  errors: z.array(z.object({
    nodeId: z.string(),
    error: z.string(),
    timestamp: z.number(),
  })).default(() => []),
  // Messages accumulated across nodes
  messages: MessagesValue,
});

// 2. Build graph from ReactFlow definition
function buildWorkflowGraph(workflowDef: WorkflowDefinition) {
  const builder = new StateGraph(WorkflowState);

  // Add each node (all are Agent nodes per D-05)
  for (const node of workflowDef.nodes) {
    if (node.type === "start" || node.type === "end") continue;
    // Agent node — wraps deepagents runtime
    builder.addNode(node.id, createAgentNodeExecutor(node), {
      retryPolicy: { maxAttempts: node.config?.retryCount ?? 1 },
    });
  }

  // Add edges from ReactFlow connections
  for (const edge of workflowDef.edges) {
    if (edge.source === "start") {
      builder.addEdge(START, edge.target);
    } else if (edge.target === "end") {
      builder.addEdge(edge.source, END);
    } else if (edge.metadata?.condition) {
      // Conditional edge — Agent output or Master Agent decides
      builder.addConditionalEdges(
        edge.source,
        createConditionalRouter(edge.metadata.condition),
        edge.metadata.targets, // e.g. { "approved": "nodeA", "rejected": "nodeB" }
      );
    } else {
      builder.addEdge(edge.source, edge.target);
    }
  }

  return builder;
}

// 3. Compile and execute
async function runWorkflow(params: {
  workflowId: string;
  projectId: string;
  triggerSource: "editor" | "chat" | "schedule";
  input: Record<string, unknown>;
}) {
  const workflowDef = loadWorkflow(params.workflowId);
  const builder = buildWorkflowGraph(workflowDef);

  // Use SQLite checkpointer (D-18) with execution-specific thread
  const checkpointer = SqliteSaver.fromConnString(CHECKPOINT_DB_PATH);
  const graph = builder.compile({ checkpointer });

  const executionId = crypto.randomUUID();
  const config = {
    configurable: { thread_id: `workflow-${executionId}` },
  };

  // Stream events to IPC channel (D-17)
  const stream = await graph.stream(
    { inputs: params.input, messages: [] },
    { ...config, streamMode: "updates" },
  );

  for await (const chunk of stream) {
    // Push to renderer via IPC: workflow:event-{executionId}
    pushWorkflowEvent(executionId, chunk);
  }
}
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| **StateGraph** | The graph builder. Takes a state schema, lets you add nodes and edges, then compile to a runnable graph. | Every workflow — this is the core entry point for building the execution graph. |
| **StateSchema + ReducedValue** | Zod-based state definition. `ReducedValue` defines how parallel branches merge their outputs (reducer function). | Define workflow state. Use `ReducedValue` for fields that fan-out nodes write to concurrently (e.g., `nodeOutputs`). |
| **START / END** | Sentinel node names. `START` is the entry point, `END` terminates execution. | Every graph needs `addEdge(START, firstNode)` and at least one edge to `END`. |
| **Send** | Dynamic fan-out. A node returns `Send[]` to dispatch parallel executions with per-branch state. | When a node needs to spawn N parallel sub-tasks (e.g., run 3 Agent nodes on different inputs). |
| **ConditionalEdgeRouter** | Function that reads state and returns the next node name (or a map for branching). | Conditional branching (D-09), loop termination (D-10), routing based on Agent output. |

### Common Pitfalls

1. **Using `addEdge` for branching — use `addConditionalEdges` instead.** `addEdge` creates a fixed, unconditional transition. If a ReactFlow edge has routing metadata (condition, loop guard), it must become `addConditionalEdges`. Forgetting this means conditional logic is silently ignored.

2. **Parallel nodes writing to the same state field without a reducer.** When multiple nodes run in parallel (fan-out/fan-in), they each return partial state updates. If a field (e.g., `nodeOutputs`) doesn't have a `ReducedValue` with a merge reducer, later writes overwrite earlier ones. Always define reducers for fields that parallel branches touch.

3. **Checkpointer thread_id namespace collision with chat runtime.** The existing `SqliteSaver` in `runtime.ts` uses `DEEPAGENT_CHECKPOINT_NAMESPACE = ''`. Workflow executions must use a distinct namespace prefix (e.g., `workflow-{executionId}`) to avoid reading/writing chat checkpoints. This is D-21: "不能替代 workflow 图级执行历史."

4. **No max iteration guard on loop edges.** LangGraph does not enforce loop limits natively. If a conditional edge routes back to a previous node (loop), you must implement a counter in state and check it in the router function. Without this, infinite loops will run until process termination.

5. **`StateSchema` vs legacy `Annotation` API.** The docs show two patterns: the newer `new StateSchema({...})` with Zod, and the older `Annotation.Root({...})`. Use `StateSchema` — it's the current API, has better TypeScript inference, and integrates with `withStructuredOutput`. The `Annotation` API is being phased out.

## 4. Implementation Guidance

**Model Configuration:**

Model selection is delegated to each Agent node's configuration (D-08: "自动继承该 Agent 的 LLM/MCP/Skills"). The workflow runtime uses `createLangChainModel()` from `src/main/deepagent/llm-adapter.ts` to instantiate the correct provider per Agent. Recommended defaults for workflow-level decisions:

| Purpose | Model | Temperature | maxTokens | Rationale |
|---------|-------|-------------|-----------|-----------|
| Agent node execution | Per Agent config (inherits from asset library) | Per Agent config | Per Agent config | D-08: no override |
| Conditional routing (Agent output decides next) | Same model as the Agent node | 0 | 256 | Routing needs deterministic output |
| Master Agent workflow tools | Project default model | 0.1 | 1024 | Tool responses need structured output |
| Loop guard / max iteration check | N/A (pure logic, no LLM) | — | — | Counter in state, no LLM call |

**Core Pattern: ReactFlow Graph to StateGraph Conversion**

The central implementation task. `graph-builder.ts` transforms a persisted ReactFlow definition into a compiled LangGraph StateGraph.

```typescript
// src/main/workflow/graph-builder.ts
import {
  StateGraph, StateSchema, ReducedValue, MessagesValue,
  START, END, Send
} from "@langchain/langgraph";
import type { GraphNode } from "@langchain/langgraph";
import { SqliteSaver } from "@langchain/langgraph-checkpoint-sqlite";
import * as z from "zod";
import type { WorkflowDefinition, WorkflowNode, WorkflowEdge } from "../../shared/types";

// --- State Schema (D-16b) ---
export const WorkflowState = new StateSchema({
  inputs: z.record(z.unknown()),
  nodeOutputs: new ReducedValue(
    z.record(z.unknown()).default(() => ({})),
    {
      inputSchema: z.record(z.unknown()),
      // Merge reducer: parallel branches each write their own nodeId key
      reducer: (existing, update) => ({ ...existing, ...update }),
    }
  ),
  routing: new ReducedValue(
    z.record(z.string()).default(() => ({})),
    {
      inputSchema: z.record(z.string()),
      reducer: (existing, update) => ({ ...existing, ...update }),
    }
  ),
  artifacts: new ReducedValue(
    z.array(z.unknown()).default(() => []),
    {
      inputSchema: z.array(z.unknown()),
      reducer: (existing, update) => [...existing, ...update],
    }
  ),
  errors: new ReducedValue(
    z.array(z.object({
      nodeId: z.string(),
      error: z.string(),
      timestamp: z.number(),
    })).default(() => []),
    {
      inputSchema: z.array(z.object({
        nodeId: z.string(),
        error: z.string(),
        timestamp: z.number(),
      })),
      reducer: (existing, update) => [...existing, ...update],
    }
  ),
  messages: MessagesValue,
});

// --- Graph Builder ---
export function buildWorkflowGraph(
  workflowDef: WorkflowDefinition,
  nodeExecutor: (node: WorkflowNode, state: z.infer<typeof WorkflowState>) => Promise<Record<string, unknown>>,
) {
  const builder = new StateGraph(WorkflowState);

  // 1. Add Agent nodes (skip start/end — they're edge sentinels, not executable)
  for (const node of workflowDef.nodes) {
    if (node.type === "start" || node.type === "end") continue;

    builder.addNode(node.id, async (state) => {
      try {
        const result = await nodeExecutor(node, state);
        // Each node writes only to nodeOutputs[nodeId] (D-14)
        return {
          nodeOutputs: { [node.id]: result },
          messages: result.messages ?? [],
        };
      } catch (err) {
        // Node failure strategy: record error, let router decide (D-11)
        return {
          errors: [{
            nodeId: node.id,
            error: err instanceof Error ? err.message : String(err),
            timestamp: Date.now(),
          }],
          routing: { [`${node.id}_status`]: "failed" },
        };
      }
    }, {
      retryPolicy: { maxAttempts: node.config?.retryCount ?? 1 },
    });
  }

  // 2. Add edges from ReactFlow connections
  for (const edge of workflowDef.edges) {
    const sourceNode = edge.source === "start" ? START : edge.source;
    const targetNode = edge.target === "end" ? END : edge.target;

    if (edge.metadata?.condition) {
      // Conditional edge (D-09: Agent output or Master Agent decides)
      builder.addConditionalEdges(
        sourceNode,
        createConditionalRouter(edge.metadata.condition, edge.metadata.maxIterations),
        edge.metadata.targets as Record<string, string>,
      );
    } else {
      builder.addEdge(sourceNode, targetNode);
    }
  }

  return builder;
}

// --- Conditional Router Factory ---
function createConditionalRouter(
  condition: string,
  maxIterations?: number,
): (state: z.infer<typeof WorkflowState>) => string {
  // Return a router function that reads routing state
  return (state) => {
    // Loop guard: check iteration counter (D-10)
    if (maxIterations !== undefined) {
      const loopKey = `__loop_${condition}`;
      const count = (state.routing[loopKey] ?? 0) as number;
      if (count >= maxIterations) {
        return "max_iterations_exceeded"; // Must be mapped to END or fallback node
      }
    }

    // Read routing decision from state
    const decision = state.routing[condition];
    if (!decision) {
      throw new Error(`No routing decision found for condition "${condition}"`);
    }
    return decision;
  };
}
```

**State Management:**

D-14 mandates the **shared state pattern**: LangGraph's State object passes between nodes. Each Agent node reads from the shared state and writes only to its own `nodeOutputs[nodeId]` key. The graph-level router reads from `routing` to make branching/loop decisions.

Key state design decisions:
- `nodeOutputs` uses `ReducedValue` with a spread-merge reducer — safe for parallel writes since each node writes a unique key
- `routing` is the control plane — Agent nodes can write routing decisions (e.g., `{ next: "approved" }`), and conditional edge routers read from it
- `errors` accumulates across all nodes — never silently swallowed
- `messages` uses `MessagesValue` (LangGraph's built-in message accumulator with deduplication)

Checkpointing uses `SqliteSaver` from `@langchain/langgraph-checkpoint-sqlite` (already installed). The checkpointer must use a **separate database file** from the chat runtime's checkpoint DB to avoid namespace collision (pitfall #3).

**Context Window Strategy:**

Each Agent node runs independently with its own LLM call — there is no single "context window" for the whole workflow. Context management is per-node:

- **Agent node context**: Inherits from Agent definition (system prompt + MCP tools). The node executor passes only relevant state slice (inputs + upstream nodeOutputs) as the user message, not the entire workflow state.
- **State pruning**: For long-running workflows with many nodes, pass only the last N node outputs (configurable, default 5) plus any explicitly tagged "important" outputs. This prevents context window overflow in downstream nodes.
- **Message accumulation**: The `messages` field in state grows across nodes. For workflows with >10 nodes, implement a summarization node that compresses message history before passing to the next Agent node.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.
> Note: This project uses TypeScript. All examples use Zod (not Pydantic) for structured outputs.

### Structured Outputs with Zod

LangGraph.js uses Zod schemas throughout: state definitions (`StateSchema`), tool parameter schemas, and LLM structured output (`withStructuredOutput`). For Agent nodes that need to return structured data (e.g., routing decisions, evaluations), define a Zod schema and use `llm.withStructuredOutput(schema)`.

```typescript
// Example: Agent node that evaluates whether to continue a loop
import * as z from "zod";
import { ChatAnthropic } from "@langchain/anthropic";

// Define the output schema
const RoutingDecisionSchema = z.object({
  decision: z.enum(["continue", "stop", "retry"]).describe(
    "Whether to continue the loop, stop, or retry with modifications"
  ),
  reason: z.string().describe("Brief justification for the decision"),
  modifications: z.record(z.unknown()).optional().describe(
    "If retrying, any parameter modifications to apply"
  ),
});

// Use withStructuredOutput for guaranteed schema compliance
const model = new ChatAnthropic({ model: "claude-sonnet-4-20250514", temperature: 0 });
const structuredModel = model.withStructuredOutput(RoutingDecisionSchema);

// In the Agent node executor:
async function evaluateLoopState(state: WorkflowState): Promise<RoutingDecision> {
  const result = await structuredModel.invoke([
    { role: "system", content: "Evaluate the current workflow state and decide next action." },
    { role: "user", content: JSON.stringify(state.nodeOutputs) },
  ]);
  // result is typed as z.infer<typeof RoutingDecisionSchema> — validated at the API boundary
  return result;
}
```

**Retry logic for structured output failures:**
- `withStructuredOutput` in LangChain.js handles parsing internally — if the LLM output doesn't match the schema, it throws a `ZodError`
- Wrap the call in a try/catch with up to 2 retries (total 3 attempts). Log the raw LLM output and the Zod validation error on each failure
- After 3 failures, surface the error to the workflow's `errors` array and set `routing[nodeId_status] = "failed"` so the graph can route to error handling
- Do NOT retry on non-validation errors (network, auth) — those should propagate immediately

### Async-First Design

LangGraph.js is fully async. Every node function is `async`, graph invocation is `await graph.invoke()` or `for await (const chunk of graph.stream(...))`.

**The one common mistake:** Calling `asyncio.run()` or equivalent inside an already-running event loop. In Electron's main process, this manifests as calling synchronous blocking code (e.g., `db.prepare().get()`) inside an async node function. `better-sqlite3` is synchronous — this is fine for quick queries, but for operations that might block (large result sets, WAL checkpoint), wrap in `await new Promise(resolve => setImmediate(() => resolve(db.prepare(...).get())))` to yield to the event loop.

**Stream vs. await:**
- Use `graph.stream()` with `streamMode: "updates"` for real-time UI feedback (D-17: IPC event streaming). Each chunk contains the state update from the node that just completed
- Use `graph.invoke()` only for programmatic calls where you need the final result (e.g., Master Agent calling `run_workflow` tool and waiting for completion)
- Never mix: don't `await graph.invoke()` and then try to stream events from a separate channel. Pick one mode per execution

### Prompt Engineering Discipline

**System vs. user prompt separation in Agent nodes:**
- The **system prompt** comes from the Agent definition's `system_prompt` field (inherited per D-08). It defines the Agent's role, capabilities, and constraints
- The **user prompt** is constructed by the node executor from the workflow state: `{role: "user", content: "Inputs: {inputs}\nUpstream outputs: {relevant nodeOutputs}\nTask: {node description}"}`
- Never merge these — the system prompt sets behavioral guardrails; the user prompt provides task-specific context

**Few-shot for routing decisions:**
- When an Agent node needs to produce a routing decision (e.g., "approved" vs "rejected"), include 2-3 examples in the system prompt: `"When output quality is high, return {next: 'approved'}. When errors exceed 3, return {next: 'retry'}."`
- Dynamic few-shot (retrieving examples from a store) is overkill for workflow routing — static examples in the prompt are sufficient

**Token budget:**
- Set `maxTokens` explicitly on every LLM call. For routing decisions: 256 tokens. For Agent task execution: 4096 tokens (or per Agent config). Never leave unbounded — a runaway generation burns cost and blocks the workflow
- The `ChatAnthropic` and `ChatOpenAI` constructors accept `maxTokens` directly

### Context Window Management

Each Agent node has its own context window. The risk is passing too much upstream state into a single node's prompt.

**Strategy for this system type:**
1. **Selective state passing:** The node executor should extract only `inputs` and `nodeOutputs[upstreamNodeIds]` relevant to the current node, not dump the entire state object
2. **Output summarization for long chains:** If a workflow has >8 nodes in sequence, implement a lightweight summarization step: after node N completes, if cumulative token count exceeds 80% of the model's context window, summarize the accumulated nodeOutputs before passing to node N+1
3. **Artifact offloading:** Large artifacts (file contents, code diffs) should be stored in the `artifacts` array with a reference ID, not inlined in `nodeOutputs`. The downstream node receives the reference and loads the artifact only if needed
4. **Message pruning:** The `messages` field should be capped at the last 20 messages per workflow execution. Older messages get summarized into a single "history" message

### Cost and Latency Budget

**Per-call cost estimate (at typical workflow scale):**
- Routing decision (structured output, 256 max tokens): ~$0.001 per call (Claude Sonnet)
- Agent node execution (4096 max tokens + tools): ~$0.01-0.05 per call depending on tool usage
- Typical 5-node workflow: ~$0.05-0.25 per execution
- At 100 workflow executions/day: ~$5-25/day

**Caching strategy:**
- **Exact-match caching:** For identical workflow + identical input, cache the final output with a TTL (e.g., 1 hour). Use a hash of `{workflowId, input}` as cache key
- **Semantic caching is NOT recommended** for workflow execution — workflows are deterministic by design; small input changes should produce fresh results

**Sub-task model routing:**
- Routing decisions and simple classifications: use the cheapest available model (e.g., `claude-haiku-4-5-20251001` or `gpt-4o-mini`) — these are structured, short-output tasks
- Agent task execution: use the model configured in the Agent definition — don't downgrade
- Workflow tools (list, status): pure database queries, no LLM needed

## 5. Evaluation Strategy

> This is a deterministic workflow execution engine, not an LLM output quality system.
> Evaluation focuses on execution correctness, state integrity, and UI-engine synchronization.
> LLM output quality is each Agent's concern (Phase 3), not the workflow engine's.

### Dimensions

| # | Dimension | Rubric | Measurement | Priority |
|---|-----------|--------|-------------|----------|
| E1 | **Graph-Execution Fidelity** | PASS: 执行 trace 中每个 step 包含 `nodeId`，step 顺序与 StateGraph 拓扑排序一致；条件分支走过的边在 ReactFlow 图中有对应 edge；循环迭代次数 <= maxIterations。FAIL: 执行走了图中不存在的路径；trace 缺少 nodeId；循环超过 maxIterations 未停止。 | Code | Critical |
| E2 | **Error Observability** | PASS: 节点失败事件包含 `nodeId`, `nodeName`, `errorType` (llm_timeout / mcp_error / state_validation / unknown), `stateSnapshot`, `retryCount`, `retryResults`；用户从错误信息可在 30 秒内定位根因。FAIL: 错误信息仅 "workflow execution failed"；缺少节点上下文；LLM 返回格式错误被静默吞掉。 | Code + Human | Critical |
| E3 | **Loop Safety** | PASS: 所有循环边在 StateGraph 编译时注入 `maxIterations` 检查；达到上限后 graph 停止并输出 `{loopTerminated: true, edgeId, iterationCount, lastOutput}`；无循环边时 router 不检查计数器（零开销）。FAIL: 循环边无 maxIterations；maxIterations 仅存 UI 配置未注入 engine；循环终止后无报告。 | Code | Critical |
| E4 | **Namespace Isolation** | PASS: workflow execution 使用 `workflow-{uuid}` 格式的 thread_id，与 chat session 的 thread_id 无交集；SqliteSaver 使用独立数据库文件或独立 namespace prefix；交叉读写测试返回空结果。FAIL: thread_id 格式与 chat session 重叠；读取到 chat checkpoint 数据。 | Code | Critical |
| E5 | **Checkpoint Recovery** | PASS: 模拟进程崩溃后，从最近 checkpoint 恢复执行，已成功节点的 nodeOutputs 完整保留不重复执行；恢复后最终 state 与无中断执行结果一致。FAIL: 恢复后从头执行；checkpoint 读取返回空或错误数据。 | Code | High |
| E6 | **Parallel Execution Consistency** | PASS: fan-out 并行节点各自写入 nodeOutputs 的独立 key (nodeOutputs[nodeA], nodeOutputs[nodeB])；fan-in 合并后所有并行结果完整存在；nodeOutputs 使用 ReducedValue spread-merge reducer。FAIL: 并行写入覆盖；fan-in 在分支未全部完成时触发。 | Code | High |
| E7 | **IPC Event Completeness** | PASS: 工作流执行全过程通过 `workflow:event-{executionId}` channel 推送事件，事件类型覆盖 node_start / node_end / node_error / workflow_start / workflow_end；每个事件包含 executionId + nodeId + timestamp。FAIL: 事件丢失；channel 未建立；事件缺少必要字段。 | Code | High |
| E8 | **State Schema Compliance** | PASS: WorkflowState 的 nodeOutputs / routing / errors / artifacts 字段类型符合 StateSchema 定义；node 只写入自己的 nodeOutputs[nodeId] key；并行 reducer 正确合并。FAIL: node 写入其他 node 的 key；state 字段类型不匹配 schema。 | Code | Medium |

## 6. Guardrails

### Online (Real-Time)

> 工作流执行引擎的在线保护。每个 guardrail 在执行路径上实时运行，延迟开销最小化。

| Guardrail | 触发条件 | 干预方式 | 实现位置 |
|-----------|---------|---------|---------|
| **循环迭代计数器** | 每次循环边被遍历时 | 达到 maxIterations 时强制路由到终止节点，输出保护报告 | `router.ts` -- ConditionalEdgeRouter 内检查 `__loop_{edgeId}` 计数 |
| **Checkpoint 写入验证** | 每次 superstep 结束后 | 写入后立即读取验证，失败时重试 1 次，仍失败则中止执行并报告 | `workflow-runtime.ts` -- checkpoint 写入后 read-back check |
| **节点错误捕获** | 每个 Agent 节点执行时 | try/catch 包裹节点执行，错误写入 state.errors，设置 routing[nodeId_status]="failed"，由后续 router 决定跳过/重试/停止 | `node-executor.ts` -- 节点执行 wrapper |
| **Namespace 前缀校验** | 执行启动时 | 断言 thread_id 匹配 `workflow-{uuid}` 格式，格式不符则拒绝执行 | `workflow-runtime.ts` -- runWorkflow 入口校验 |

### Offline (Flywheel)

> 离线质量信号，采样分析驱动改进循环。

| 指标 | 采样策略 | 降级时的动作 |
|------|---------|-------------|
| **执行成功率** | 每次执行记录到 workflow_executions 表 | 成功率 < 90% 时：检查 top 失败节点，review 错误类型分布 |
| **节点平均执行时长** | 每次执行记录每个 node_run 的耗时 | 某节点 P95 > 60s 时：检查是否 LLM 超时或 MCP 阻塞，考虑增加默认超时 |
| **循环保护触发率** | 每次循环终止记录到执行日志 | 触发率 > 20% 时：review 循环条件设计，可能需要调整默认 maxIterations |
| **Checkpoint 恢复成功率** | 记录每次恢复尝试的结果 | 恢复失败率 > 5% 时：检查 SqliteSaver 写入逻辑和 DB 文件完整性 |

**Smart Sampling Strategy:**

|---------|--------|---------|
| 执行失败 (status = failed) | 最高 | 全量记录，含完整 state snapshot + error trace |

| 执行时长 > P95 | 中 | 记录各节点耗时分布 |
