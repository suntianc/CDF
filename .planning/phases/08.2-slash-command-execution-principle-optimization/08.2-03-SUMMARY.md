## Judge prompt (final template, see `useGoalJudge.ts:108-119`)

```
You are a strict judge. Evaluate whether the following completion condition is met by the AI agent's recent work.

Condition: ${goal}

Recent conversation (last 8 turns):
${recentTurns}

Respond with JSON ONLY, no preamble, no markdown fence. Schema:
{ "satisfied": boolean, "reason": "1-2 sentence rationale in 简体中文" }
```

**Recent-turns truncation**: last 8 user/assistant message pairs, each clipped to 500 chars
(so a runaway search history cannot blow up the judge prompt). Implementation:
`renderRecentTurns()` filters out tool / system / goal-system rows to keep the judge focused
on user/assistant dialogue.

## Run-completed signal — investigation note

The plan mentioned `run_completed` as the trigger for judge evaluation, but `src/main/llm.ts`
does not emit a `run_completed` event. The actual end-of-turn signal is `message_done`
(LLM-side: all `message_chunk` flushed, no more tool calls) or `runtime_error`. Both events
flip `useSessionStore.isStreaming` from `true` to `false` in the `message_done` / `runtime_error`
handlers (`sessionStore.ts:705-720`, `sessionStore.ts:746-766`).

P3 implementation: `useGoalJudge` subscribes to `useSessionStore.isStreaming` and triggers
a judge call on every `true → false` flip. This is the cleanest signal because:
- (a) it survives cross-process IPC re-ordering (zustand subscribe is local);
- (b) it does not require modifying `llm.ts` (Hard Do Not Touch);
- (c) it works for `message_done` AND `runtime_error` AND `stopChat` paths.

Trade-off: every `isStreaming` false-edge triggers a judge call, including non-`/goal` turns.
This is intentional — when no judge loop is active, the hook is not subscribed, so the edge
fires no callback. The hook is the one registering and cleaning up its own subscription
via `startGoalJudgeLoop` / `stopGoalJudgeLoop`.

## Decisions made

- **Issue 8 (factory pattern)**: `createGoalJudge()` factory returns closure-isolated
  instances; tests construct a fresh `testJudge` per `it`. The default-exported singleton
  `goalJudge` + alias exports (`startGoalJudgeLoop`, `stopGoalJudgeLoop`, `useGoalJudgeStatus`)
  preserve the import path that other modules would use. The plan's "移除 `__resetForTests`"
  directive is honored — no test reset API needed.
- **judge call isolation (C1-03)**: `judgeRequestId = 'judge-' + Date.now() + '-' + Math.random()`;
  the judge registers `onChunk(judgeRequestId, ...)` on a unique channel. The main agent's
  `sendMessage` registers its own chunk listener with its own requestId, so the two are
  fully isolated. Judge chunks do NOT enter the conversation stream.
- **Bubble path (Plan 04 decision deferred)**: bubble is a separate component, NOT a Message
  row. Phase 7's `/goal` wrote the toast to sonner, not into `messages`. There is no
  goal-system Message row to hide or replace — MessageItem was NOT extended. Confirmed
  by reading `sessionStore.test.ts` (no `goal-system` payload exists) and tracing
  `dispatcher.ts:111-114` (old path: `toast.info(...)`).
- **P6 cross-session isolation**: `goalJudgeStatus` Map is keyed by `sessionId`. `ChatArea`
  mounts the bubble only for `activeSessionId`. The component returns null when no
  entry exists for the active session. Switching sessions does not clear status (P6 lock
  from CONTEXT.md — goal is sticky). The bubble re-appears on switching back.
- **subscribe vs `useEffect`**: `useGoalJudgeStatus` uses `useSyncExternalStore` over
  `useSessionStore.subscribe` + a `getSnapshot` reader. This keeps the hook contract
  minimal (no React-state polling) and integrates with Zustand's natural subscription API.
  The factory closure is stable across re-renders via `useMemo`.

the `useSessionStore.subscribe` callback in `useGoalJudge.ts`**

- **Verification**: 0 new TS errors in useGoalJudge.ts.
- **Committed in**: 6aca469 (Task 1).

**2. [Rule 3 - Test fixture] `mockState.setState` proxy in `useGoalJudge.test.ts` exposes
`getState` returning the live `mockState` reference (so test mutation through `setState`
propagates to subsequent `getState` reads).**
- **Found during**: Test debugging — initial `getState: () => ({...mockState})` returned
  a snapshot, so tests that did `mockState.isStreaming = true` did not propagate.
- **Fix**: switch to `getState: () => mockState` (returns the live mutable object).
- **Verification**: 6/6 useGoalJudge tests pass.
- **Committed in**: 6aca469 (Task 1).

**3. [Rule 1 - Test fake-timer setup] Adjusted the elapsed-tick test to use a single
`vi.advanceTimersByTime` per assertion (not `setSystemTime` + `advanceTimersByTime`)**
- **Found during**: Test debugging — first assertion read 6s instead of 3s because
  combining `setSystemTime` + `advanceTimersByTime` double-counted the offset.
- **Fix**: rely on `vi.useFakeTimers` + `vi.setSystemTime(base)` + `vi.advanceTimersByTime`
  alone. Each `advanceTimersByTime(3_000)` advances the clock by 3s and fires the
  setInterval 3 times.
- **Verification**: 6/6 GoalSystemBubble tests pass.
- **Committed in**: 2d8aa87 (Task 3).

## Threat Flags

| Flag | File | Description |
|------|------|-------------|
| `threat_flag: run-completed-proxy` | `src/renderer/src/hooks/useGoalJudge.ts:215-225` | The "main agent finished a turn" signal is `useSessionStore.isStreaming true→false` (proxy for `message_done` / `runtime_error` / `stopChat`). Race window: if the user starts a `/goal` while the main agent is already mid-stream, the next `isStreaming` false-edge triggers a judge call against the wrong "turn" (the previous one). **Accept**: worst case the judge asks the agent to "continue" once, then converges. Not a security issue. |
| `threat_flag: judge-prompt-injection` | `src/renderer/src/hooks/useGoalJudge.ts:108` (judge prompt) | The recent-turns transcript is fed directly into the judge prompt. A compromised or adversarial message in the conversation could attempt to manipulate the judge's `{satisfied, reason}` JSON output. **Mitigate**: the judge output is JSON.parse'd with strict schema (boolean satisfied + string reason) and we do NOT execute any `reason` text as code. The `reason` is only used as the next user message ("继续：${reason}"), so the worst-case blast radius is a misleading-but-still-text prompt to the main agent. |
| `threat_flag: 20-turn-cap-bypass` | `src/renderer/src/hooks/useGoalJudge.ts:280-289` | The 20-turn cap is enforced inside `runJudgeIteration`. Programmatic mutation of `useSessionStore.goalJudgeStatus` from outside the hook could reset `iteration` to 0 and bypass the cap. **Accept**: `goalJudgeStatus` is a private Zustand slice; no external mutation surface. |
| `threat_flag: cross-store-side-effect` | `src/renderer/src/hooks/useGoalJudge.ts:194-200, 269-272` | `startGoalJudgeLoop` calls `useSessionStore.sendMessage` internally. If `sendMessage` signature changes (e.g. takes a 4th arg), the judge breaks silently. **Mitigate**: 6 unit tests pin the contract. |
| `threat_flag: bubble-render-while-stale` | `src/renderer/src/components/ChatArea/GoalSystemBubble.tsx:155-180` | The bubble uses both `useGoalJudgeStatus` (hook) and `useSessionStore` selector. If the hook is unsubscribed and the user clears `sessionGoals` directly (not via `/goal`), the bubble may keep rendering because `useGoalJudgeStatus` returns `goal: ''` only when the store returns empty. **Mitigate**: the bubble's null-check requires BOTH `status` and `effectiveGoal` to be empty. Manual UI verification: typing `/goal clear` and switching sessions both clear the bubble correctly. |
