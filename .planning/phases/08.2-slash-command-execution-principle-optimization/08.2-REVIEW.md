### WR-01: `useGoalJudgeStatus` 的 `getSnapshot` 每次返回新对象，违反 `useSyncExternalStore` 契约

**文件：** `src/renderer/src/hooks/useGoalJudge.ts:328-336`

**严重度：** 警告

**描述：** `getSnapshot` 在每次调用时构造一个新对象 `{ entry, goal }`，而 React 的 `useSyncExternalStore` 期望 `getSnapshot` 在底层状态未变时返回**稳定引用**（通过 `Object.is` 比较）。当前实现在 `getSnapshot` 内重新构造对象会导致：
1. React 在 dev/strict 模式下触发「The result of getSnapshot should be cached」警告；
2. 任何 `useSessionStore` 状态变化都会让 `GoalSystemBubble` 重渲染（即使 `entry` / `goal` 实际未变）。

```ts
// 当前实现
const getSnapshot = () => {
  if (!sessionId) return { entry: undefined, goal: '' };
  const s = useSessionStore.getState();
  return { entry: s.goalJudgeStatus.get(sessionId), goal: s.sessionGoals.get(sessionId) ?? '' };
};
```

测试 `useGoalJudge.test.ts` 6 个 case 均直接调用 `startGoalJudgeLoop`/`stopGoalJudgeLoop`，**未通过 `renderHook` 触发 `useGoalJudgeStatus`**，所以这个问题未在测试中暴露。

**建议修复：** 改用 `zustand` 自带的 selector 订阅，或用 `useSyncExternalStoreWithSelector`（react 18 内置）；最简方案是直接用 `useSessionStore((s) => ({ entry: s.goalJudgeStatus.get(sessionId), goal: s.sessionGoals.get(sessionId) ?? '' }))` 走 Zustand 的浅比较（Zustand v5 已原生支持 `useSyncExternalStore`，并会做引用稳定化）。

```ts
let placeholderWarned = false;
```

### WR-03: `useGoalJudgeStatus` 钩子的"工作中"工作时间 ticker 状态初始化不一致

**文件：** `src/renderer/src/components/ChatArea/GoalSystemBubble.tsx:122-135`

**严重度：** 警告

**描述：** 组件用 `useState` 初始化 `tickElapsed`，但 `useEffect` 立刻再次 setState 到相同值（首次挂载时），然后启动 `setInterval`。这导致：
1. 首次渲染与 `useEffect` 后第一次状态更新之间存在一次「无用 re-render」；
2. `setInterval` 回调内每次 `setTickElapsed(...)` 都读 `Date.now() - startedAt`，但 `startedAt` 是闭包变量，在 `useEffect` 依赖里已固定；如果 `startedAt` 在 hook 后续轮次中变化（典型情况：`useGoalJudgeStatus` 跨 session 切换），需要清理 + 重建 interval（当前依赖 `[isWorking, startedAt]` 正确处理了，但 `useState` 初始化阶段没用 `useState(() => ...)` 工厂函数读 `startedAt`，导致 SSR 或挂载时 `Date.now() - 0` 计算出 NaN-then-0）。

实际不会崩溃（fallback 逻辑存在），但 `useState(() => ...)` 工厂函数 + 初始 `Date.now() - startedAt` 更合理，且能省一次无用 re-render。

**建议修复：** 把 `useState` 改成惰性初始化，并在 effect 中省略第一次同步 setState：

```ts
const [tickElapsed, setTickElapsed] = useState<number>(() =>
  startedAt ? Math.floor((Date.now() - startedAt) / 1000) : 0
);
useEffect(() => {
  if (!isWorking) return undefined;
  setTickElapsed(startedAt ? Math.floor((Date.now() - startedAt) / 1000) : 0);
  const id = setInterval(() => {
    setTickElapsed(startedAt ? Math.floor((Date.now() - startedAt) / 1000) : 0);
  }, 1000);
  return () => clearInterval(id);
}, [isWorking, startedAt]);
```

### WR-04: `startGoalJudgeLoop` 在主 agent 正在 streaming 时被静默丢弃

**文件：** `src/renderer/src/hooks/useGoalJudge.ts:200-212` + `src/renderer/src/stores/sessionStore.ts:412-414`

**严重度：** 警告

**描述：** 当用户在前一个流还没结束时输入 `/goal X`，dispatcher 会调 `startGoalJudgeLoop(sessionId, goal)`，后者执行 `await useSessionStore.getState().sendMessage(projectId, goal)`。但 `sendMessage` 在第 414 行有早退守卫：

```ts
sendMessage: async (projectId, content, overrides?) => {
  const { activeSessionId, isStreaming, sessions } = get();
  if (!activeSessionId || isStreaming) return;  // 静默 return
```

结果是 goal 没有作为 user message 注入，但 `setGoalJudgeStatus('idle')` 已经在前一行执行，订阅了 `isStreaming` 转换。当流结束时（prev streaming false 翻转），judge 启动并试图解析一个空对话流 → 必然返回 `satisfied: false` → 然后注入「继续：...」但又被 `isStreaming` 卡住 → 死循环直到 20 轮上限。

**建议修复：** 要么 `startGoalJudgeLoop` 自身在 `useSessionStore.getState().isStreaming` 为 true 时排队等待（订阅 isStreaming 翻转后再注入），要么在 dispatcher 入口检测 `isStreaming` 并提示用户先停止生成。最小修复：在 `startGoalJudgeLoop` 注入失败时立刻 `setGoalJudgeStatus({ status: 'failed', reason: '主 agent 正在生成，请等待完成后再 /goal' })`。

### WR-05: `PlanPopup` "修改计划" 按钮在 `iterationCount === 20` 但 `status !== 'cap-reached'` 时仍可点击

**文件：** `src/renderer/src/components/PlanPopup/PlanPopup.tsx:40, 132-138` + `src/renderer/src/stores/planPopupStore.ts:94-96`

**严重度：** 警告

**描述：** `isCapReached = status === 'cap-reached'` 控制按钮 disabled。但 `startModify` 在第 21 次点击时才把 `status` 设为 `cap-reached`（store 内部拒绝路径）。这意味着：

```ts
const isCapReached = status === 'cap-reached' || iterationCount >= MAX_ITERATIONS;
// ...

```

或者在 store 中 `startModify` 拒绝时同步把 `iterationCount` 回滚一次（保持语义"20 次后真的不能改了"）。

### WR-06: PlanPopupStore close 内的 `stopChat` 是 fire-and-forget，可与后续 open() 竞态

**文件：** `src/renderer/src/stores/planPopupStore.ts:67-76`

**严重度：** 警告

**描述：**

```ts
close: () => {
  const requestId = get().currentRequestId ?? useSessionStore.getState().streamingMessageId;
  if (requestId) {
    window.electronAPI?.llm?.stopChat(requestId)?.catch((err) => { ... });
  }
  set({ ...INITIAL_STATE });
},
```

`stopChat` 触发后立即 `set({ ...INITIAL_STATE })`，不 await。如果用户瞬间关 + 开（理论可能），新弹窗的 `currentRequestId` 还没来得及被 stop 取消。P11 风险已部分缓解（store 状态先清空），但 IPC 层的 stop 是异步的，主进程可能仍往旧 requestId 推流（且新 open() 内的 `appendChunk` 会因为 `currentRequestId` 被新值覆盖而不会写到旧 ID）。

实际后果有限（orphan 流会自己结束），但代码可读性差，且 P11 pitfall 文档指明 "startModify 先调 cancelCurrent" — close 也应 await 自己的 stopChat。

**建议修复：** close 改为 async 并 await `stopChat`，或在 store action 上加 `close: () => Promise<void>` 签名让组件 `await`：

```ts
close: async () => {
  const requestId = get().currentRequestId ?? useSessionStore.getState().streamingMessageId;
  if (requestId) {
    try { await window.electronAPI.llm.stopChat(requestId); }
    catch (err) { console.warn('[planPopupStore] stopChat during close failed:', err); }
  }
  set({ ...INITIAL_STATE });
},
```

### IN-01: `commands:readBody` handler 在路径校验后存在 TOCTOU（`fs.existsSync` → `fs.readFileSync`）

**文件：** `src/main/ipc-handlers.ts:848-853`

**严重度：** 信息

**描述：** 路径白名单校验通过后，handler 用 `fs.existsSync(resolved)` 检查文件是否存在，再 `fs.statSync` + `fs.readFileSync`。两次 fs 系统调用之间存在微观 TOCTOU 窗口 — 攻击者可在 `existsSync` 通过后用符号链接替换为任意文件。实际风险极低（白名单限制了 `~/.cdf/commands/` 和 `<projectPath>/.cdf/commands/`，且每次调用 bodyPath 都来自 renderer 端 SlashCommand.bodyPath，path 来自 `command-registry` 阶段解析）。ASVS V5.1.3 的核心保护（路径白名单 + 类型 + 长度）已到位。

**建议修复：** 删掉 `fs.existsSync` 单独调用，让 `fs.statSync` 自己抛 `ENOENT`（在 try-catch 内已处理）；或用 `fs.promises.stat(resolved, { throwIfNoEntry: false })` 单次调用。
