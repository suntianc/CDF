# Phase 08.2: Slash command execution principle optimization - Context

**Gathered:** 2026-06-05
**Status:** Ready for planning

<domain>
## Phase Boundary

基于 Claude Code + Codex 桌面版调研，重构 CDF 4 个核心 slash command 的「执行原则」：

1. **`.cdf/commands/*.md` body 真正参与 dispatch** — 现状 dispatcher 完全不读 .md body，直接硬编码 prompt 模板（Phase 6 D-18 简化）
2. **`/goal` 真正"驱动" agent** — 现状 goal 写进 store 后 LLM 完全不知道（v1.1 in-memory 是写后即忘）
3. **`/context` 完整化** — 现状只显示 4 项 token breakdown，远不如 Claude Code 详细
4. **`/plan` 走 Codex 弹窗模式** — 现状一次性 LLM 调用 + toast 反馈，无 plan 审阅 + modify loop
5. **完整 frontmatter 字段执行** — 现状只解析 3 个字段，Claude Code 十几种字段全部未实现

**不在范围**：
- ❌ MCP 命名维度变更（保持 per-server，B1 决定）
- ❌ Plugin dispatch 绕过 LLM 的 direct call 改造（G1 决定：LLM 始终在 loop）
- ❌ /plan 持久模式（采用 plan-then-execute 弹窗模式，不做持久权限模式）
- ❌ 轻量级 token 估算改 gpt-tokenizer（v1.2+ SLASH-15 范围）
- ❌ Args 解析层（v1.2+ 推）
- ❌ SLASH-15 (`/goal` SQLite 持久化 — v1.2+ 推)
- ❌ SLASH-17 (命令别名 — v1.2+ 推)

</domain>

<decisions>
## Implementation Decisions

### A. .md body 加载 + $ARGUMENTS 替换

- **D-01 (加载时机):** **Lazy load on dispatch** — 按 Enter 时才读 .md 文件。简单、与 A1 一致。
- **D-02 (替换语法):** **完整 Claude Code 对等** — 支持 `$ARGUMENTS`（整串）、`$0`/`$1`/`$2`（按位置取参数）、`$name`（命名参数，需在 frontmatter `arguments: [env, flags]` 声明）。
- **D-03 (注入方式):** **body 替换后作为 user message 代替原命令** — 命令名不出现。`/deploy production 顺带更新 changelog，谢谢` 整段作为 args 替换。
- **D-04 (文件丢失处理):** **chokidar 主动监控 + LLM 兜底** — `unlink` 事件触发从 registry 移除。竞态场景下 LLM 收到"命令不存在"消息兜底。**不**弹用户 toast。
- **D-05 (扩展 SlashCommand):** 新增字段 `bodyPath: string`（指向 .md 文件路径），让 dispatcher 在 dispatch 时调 IPC 读 body。
- **D-06 (新 IPC):** `commands:readBody(bodyPath: string): Promise<{ body: string; mtimeMs: number }>` — 返回 body 内容 + mtime（用于 cache 失效判断）。

### B. MCP 命名维度（保持现状）

- **B-01:** 保持 **per-server 现状**（一个 slash 命令 = 一个 MCP 服务器），不展开为 per-tool。LLM 负责从该 server 选工具。

### C-1. /goal 理想形态（"驱动 agent 直到完成"模式）

- **C1-01 (完成判定):** **专用内置 agent（用户不可见）+ 复用当前对话模型** — 主 agent 完成一轮时说"做完了"才调 judge agent，judge 返回 yes/no + 理由。
- **C1-02 (goal 注入):** **D 方案** — 首次告诉一次目标（作为 user 消息）+ judge 返回 no 时把理由作为下轮指引（也是 user 消息）。对标 Claude Code Stop hook。
- **C1-03 (judge 触发):** **主 agent 说"做完了"时才调** — 不是每轮 tool 调完都判。
- **C1-04 (UI 呈现):** **系统气泡（实时状态更新）** — 设置后气泡显示「◎ 目标：X 状态：工作中 耗时：1m 23s」；完成时变「✅ 达成：X」。
- **C1-05 (judge 实现):** 在 `dispatcher.ts` 加 `PlanMode`-like 第五种 kind: `GoalLoop`。内部启动新 `runLLMChat` 调用（用同一 model），传入 condition + 最近 N 轮对话，judge 返回 `{ satisfied: boolean; reason: string }`。

### C-2. /context 理想形态（Claude Code 完整版 + CDF 专属项）

- **C2-01 (数据范围):** **Claude Code 完整版 + CDF 专属项** —
  - 必含：模型名 + 总量 (X/Y + %) + 进度条 + 详细分类 + Free space + Autocompact buffer (CDF 85% 阈值) + 单个 MCP tool 明细
  - CDF 专属项：Workflows、Project command bodies（被 lazy load 后的 .md 内容占多少 token）
- **C2-02 (UI 呈现):** **弹窗 modal** — 桌面应用不污染聊天流。
- **C2-03 (LLM 可见性):** **LLM 不知道**（现状）— /context 是给用户看的工具数据，不是 LLM 的指令。
- **C2-04 (触发方式):** **slash 命令 + 按钮双入口** — `/context` 命令保留，同时增加一个常驻按钮（📊 查看 context）触发同一 modal。

### C-3. /plan 理想形态（Codex 弹窗模式 + modify loop）

- **C3-01 (形态):** **Codex 桌面版风格** — `/plan X` 后弹窗显示计划 + 「修改 / 立即执行」按钮 + modify loop。**不**做持久模式。
- **C3-02 (计划生成):** **planOnly flag + 工具分级** — planOnly=true 期间，**写类工具屏蔽**（Edit / Write / Bash 写操作），**读类工具允许**（Read / Grep / Glob / list files）。LLM 可以探索代码库，但**不能动手**。
- **C3-03 (执行触发):** **模拟用户消息「立即执行」**（Codex 风格）— 点击执行按钮后，系统把"立即执行"作为新 user 消息塞进对话。LLM 据此进入执行（下次 sendMessage 自然不带 planOnly flag）。
- **C3-04 (modify loop):** **循环「修改意见」+ 重新 planOnly** — 每次修改意见作为新 user 消息 + planOnly=true 重发，LLM 更新计划。直至用户点"立即执行"或关弹窗。
- **C3-05 (弹窗入口):** 弹窗用 Radix Dialog（或 shadcn Dialog），状态管理用 Zustand store。`PlanMode` dispatch kind 增加 `popupOpen: true` 标志。

### D. Frontmatter 字段执行

- **D-07 (字段选择):** **完整 Claude Code 对等** — 实现 4 个核心字段：
  - `disable-model-invocation: bool` — true 时从 LLM 上下文完全移除该 skill 描述
  - `user-invocable: bool` — false 时从 / popup 隐藏
  - `allowed-tools: string[]` — skill 激活时这些工具自动批准
  - `when_to_use: string` — 追加到 description 后，让 LLM 据此判断何时自动调
- **D-08 (解析机制):** **frontmatter 预读 + body lazy** — Registry 构建时读 .md frontmatter（小、便宜），body 按需 lazy load。对标 Claude Code 设计。
- **D-09 (生效机制):** **全代码层（对标 Claude Code）** —
  - `disable-model-invocation`: 硬移除 skill 描述从 LLM 上下文
  - `user-invocable`: UI 层 popup 过滤
  - `allowed-tools`: 工具池白名单（运行时硬限制）
  - `when_to_use`: 进 system 提示（LLM 自觉）
- **D-10 (默认值):** **对齐 Claude Code** — `disable-model-invocation: false`，`user-invocable: true`，`allowed-tools: []`（所有工具可用），`when_to_use: ""`（空）。

### E. Goal 注入（已合入 C-1）

- E 不单独实现 — C1-02 (D 方案) 已涵盖 goal 注入：首次告诉 + judge 理由。
- 备选方案「每轮都念」被否决（token 浪费），「不告诉」被否决（写后即忘）。

### F. 真实气泡反馈（按 command 区分）

- **F-01 (按 command 选呈现):** 不同 system command 用最适合的呈现方式：
  - `/goal` → 系统气泡（持续更新，状态机）
  - `/context` → 弹窗 modal（不占聊天位）
  - `/plan` → Codex 风格弹窗（计划展示 + modify loop）
  - **不**统一为一种呈现（3 个 command 用途不同）

### G. Plugin dispatch direct tool call（无需变更）

- **G-01:** **不需架构变更** — LLM 始终在 loop 里：
  - Skills / Workflows / Project commands: 现状（prompt-based，LLM 编排）
  - MCP: per-server + LLM 从 server 选工具（现状）
- 理由：MCP 斜杠命令 = 告诉 LLM "用这个 MCP"，**MCP 内部工具怎么用是 LLM 的事**，我们不操心。

</decisions>

## Specific Ideas

### D-03 用户栗子（$ARGUMENTS 替换）
- 用户输 `/deploy production --force 顺带更新 changelog，谢谢`
- 假设 `deploy.md` body: "请部署到 **$0** 环境。参数：**$ARGUMENTS**。附加要求：$ARGUMENTS"
- 替换后 user message: "请部署到 **production** 环境。参数：**production --force 顺带更新 changelog，谢谢**。附加要求：production --force 顺带更新 changelog，谢谢"
- 关键：**整段中文补充语都被当作 args 一次性替换**

### C-1 /goal judge 实现路径
- `useGoalJudge` hook 内部启动新 `runLLMChat` 调用
- judge prompt: "以下 condition: `${goal}` 是否满足？基于最近对话：${lastNTurns}。返回 JSON { satisfied: bool, reason: string }"
- main agent 完成一轮后，session store 收到 "I'm done" 信号 → 触发 judge
- judge 返回 no → main agent 收到 user 消息「继续：${reason}」→ 继续干活
- judge 返回 yes → goal bubble 状态变 ✅

### C-2 /context modal 排版（参考用户上传截图）
- 顶部：模型名 + 总量 (51.5k/200k tokens 26%) + 视觉进度条
- 中部：分类 breakdown
  - System prompt: 9.5k (4.8%)
  - System tools: 17.7k (8.8%)
  - MCP tools: 11.8k (5.9%) ← 包含 per-tool 明细
  - Workflows: 7.9k (4.0%) ← CDF 专属
  - Custom agents: 7.9k (4.0%)
  - Memory files: 1.3k (0.7%)
  - Skills: 3.1k (1.5%)
  - Messages: 177 (0.1%)
  - Free space: 115.5k (57.8%)
  - Autocompact buffer: 33k (16.5%) ← CDF 85% 阈值
- 底部：MCP tools 明细（mcp__codegraph__codegraph_callees: 154 tokens ...）

### C-3 /plan 弹窗排版（Codex 风格）
- 弹窗：Radix Dialog 从顶部降下
- 内容区：LLM 生成的计划文本（markdown 渲染）
- 右下角：两按钮「修改计划」「立即执行」
- 修改模式：内嵌 input + send，循环 modify loop
- 立即执行：弹窗关闭，模拟 user 消息「立即执行」入对话流

### D-04 chokidar unlink 处理
- `chokidar-watcher.ts:watchProjectCommandsDir` 已有 unlink 监听
- 新增：unlink 时**先**调用 `command-registry.removeByPath(path)` 移除命令 + `commands:changed` IPC 推送
- 不**不**给用户弹 toast（避免打扰）

### D-09 frontmatter 字段硬强制实现路径
- `disable-model-invocation: true` → registry 构建时不把该 skill 的 description 加进 LLM 的「available skills」列表
- `user-invocable: false` → SlashCommandPopup 过滤不显示
- `allowed-tools: ['Read', 'Grep']` → dispatch 时改写 `payload.overrides.allowedTools`
- `when_to_use: '用户询问 X 时'` → description 字段追加这段文字
