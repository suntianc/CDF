## Summary

Phase 08.2 is **5 distinct workstreams** all anchored to v1.1 shipped infrastructure. The hard constraints are: `runLLMChat` (src/main/llm.ts:304-425) is **Hard Do Not Touch** (6-hunk patch-package on `@langchain/anthropic@1.4.0` lives at `patches/@langchain+anthropic+1.4.0.patch` with 6 hunks); `payload.overrides.planOnly` is the **only** LLM extension seam; `SlashCommand` and `CommandDispatchAction` are the type extension seams. Everything else is incremental.

The 5 workstreams are largely orthogonal in implementation but share 3 cross-cutting risks: (1) hard-runtime integration with the M3 thinking chain (P2 — judge agent must preserve patch-package behavior); (2) backslash-injection into tool schemas (P7 — `$ARGUMENTS` substitution is a string template, not args parse); (3) cross-platform chokidar unlink semantics (D-04 unlink on Windows uses polling).

**Primary recommendation:** Decompose into 4-5 plans, one per workstream, sequenced: **(P1) A.md body + frontmatter** (foundation) → **(P2) C-3 /plan popup** (extends existing `PlanMode` kind, lowest risk) → **(P3) C-1 /goal judge** (adds new `GoalLoop` kind, hard-runtime risk) → **(P4) C-2 /context modal** (UI-only, lowest new-code risk) → **(P5) [optional] cross-workstream polish + button + integration test**. Plan 1 must land first because D-05/D-06 IPC + body loading are referenced by C-1 (project command body tokens in /context) and the frontmatter fields gate registry-side filtering.

**Key design decision:** Current `parseFrontmatter()` in `project-commands.ts:27-46` is a hand-rolled minimal parser. It only handles `key: value` per line. D-07's 4 fields include:
- `disable-model-invocation: bool` — needs to parse `true`/`false` (current parser returns string)
- `user-invocable: bool` — same
- `allowed-tools: string[]` — needs YAML array support (current parser would store the raw `[Read, Grep]` string)
- `when_to_use: string` — single string (current parser handles this)
- `arguments: [env, flags]` — YAML list (D-02)

**Recommendation:** Replace hand-rolled parser with `yaml@2.9.0` (zero deps, single-file, ~150KB) — **MEDIUM confidence** based on training data + npm registry. The new `parseFrontmatter` should return a typed `ParsedFrontmatter` interface. This is a small, surgical change to a single function and is the cleanest fix.

**Alternatives considered:**
- Extend hand-rolled parser: brittle; would need special handling for booleans, arrays, multi-line strings. Reject.
- `gray-matter@4.0.3` (npm verified): more full-featured but pulls in `section-matter`, `strip-bom-string`, etc. Overkill for 4 fields. Mark as `gray-matter` **[WARNING: slopcheck flagged as suspicious — verify before using.]** if used; otherwise prefer `yaml`.
- `js-yaml`: pre-existing common pick; verify `npm view js-yaml version` before installing.

// 1. User types `/plan X` → dispatcher.dispatch({kind: 'PlanMode', popupOpen: true})

### Anti-Patterns to Avoid

- **Modifying `runLLMChat`**: would invalidate 6-hunk patch-package + SLASH-REGRESSION tests. Reject.
- **Modifying `llm-adapter.ts`**: same. Reject.
- **Adding a 6th dispatch kind path to LLM** (e.g. `payload.overrides.goalLoop: true`): would touch `runLLMChat`. Use a separate `runLLMChat` call (C1-05 D-09).
- **Bubbling judge agent events as separate user-visible bubbles**: judge is internal (C1-01). Inject judge reason as a **regular user message** (C1-02 D-09), not a new bubble type.
- **Catching chokidar unlink at the renderer level**: chokidar is in main process (chokidar-watcher.ts). Main must call IPC bridge to push `commands:changed`; renderer reacts via existing `onChanged` listener.
- **Implementing `$ARGUMENTS` as a flag parser**: D-02 says raw string substitution. Flag parsing deferred (CONTEXT.md deferred ideas).
- **Adding `yaml` as a dep if `gray-matter` already in tree**: `gray-matter` is **NOT** in deps (verified `package.json`); `yaml` is the right call. Mark `yaml` `[ASSUMED]` until install verification at plan time.
- **Modifying `parseFrontmatter` to keep both hand-rolled + yaml paths**: the hand-rolled parser is too brittle for 4 new fields. Replace entirely.

## Common Pitfalls

### Pitfall P1: 6-hunk patch-package regression
**What goes wrong:** Accidentally modifying `src/main/llm.ts:306-425` or `src/main/deepagent/llm-adapter.ts` breaks the `@langchain/anthropic@1.4.0` patch — 3 SLASH-REGRESSION it blocks in `src/main/deepagent/llm-adapter.test.ts` will fail.
**Why it happens:** 08.2 touches many files in the LLM path (dispatcher, payload.overrides). Refactoring temptation.
**How to avoid:** Mark both files as `// HARD DO NOT TOUCH` in plan 1. Add to `/gsd:verify-work` check.
**Warning signs:** SLASH-REGRESSION fails at `/gsd:verify-work`. Diff touches `llm.ts` or `llm-adapter.ts`.
**Source:** `patches/@langchain+anthropic+1.4.0.patch` (6 hunks verified), `src/main/deepagent/llm-adapter.test.ts` (Phase 7 D-16..D-19).

### Pitfall P2: Judge agent breaks M3 thinking chain
**What goes wrong:** Judge agent call goes through `runLLMChat`; if the judge prompt streams reasoning but the patch-package extraction is somehow disabled, the `<think>` block might emit before the JSON body, confusing the parsing.
**Why it happens:** The judge expects JSON output but LLM might first emit thinking (M3 chain preserves thinking via patch-package).
**How to avoid:** Strip <think> blocks from judge output before JSON parsing. Use `accumulator.takeReasoning()` to capture reasoning separately (already in `src/main/llm.ts:242-246`).
**Source:** Phase 7 PITFALLS P2, `src/main/llm.ts:340-403` (reasoning handling).

### Pitfall P3: $ARGUMENTS injection into tool schema (P7 in CONTEXT)
**What goes wrong:** If dispatcher does `toolName.call({ args })` instead of injecting args into the LLM prompt, a crafted `/deploy "production; rm -rf /"` could inject shell. This is **PITFALLS P7**.
**Why it happens:** Natural temptation to treat body as args.
**How to avoid:** D-03 is the fix: **body replaces user message**, never tool args. Plugin commands continue to use `args.trim()` as natural-language context (existing `dispatcher.ts:61-63`).
**Source:** CONTEXT.md PITFALLS P7, existing `dispatcher.ts:150-155` (PluginRewrite path).

### Pitfall P4: chokidar unlink cross-platform
**What goes wrong:** Windows uses `usePolling: true` by default; unlink semantics differ from macOS. Atomic writes on Windows (e.g., VSCode save) may fire unlink + add sequences.
**Why it happens:** Chokidar normalizes most events but Windows behavior is non-deterministic.
**How to avoid:** Keep `awaitWriteFinish: { stabilityThreshold: 200 }` (already in `chokidar-watcher.ts:162`); debounce 100ms (already at line 173). Test cross-platform via macOS + Windows + Linux matrix (CONTEXT.md D-04 references Phase 6 D-25).
**Source:** `src/main/commands/chokidar-watcher.ts:160-188`, CONTEXT.md D-04.

### Pitfall P5: Popup focus loss
**What goes wrong:** Opening `/plan` popup steals focus from composer; user can't type modify opinion.
**Why it happens:** Radix Dialog's default focus trap.
**How to avoid:** Use Radix `Dialog` with `onOpenAutoFocus` to focus the modify input first. `src/renderer/src/components/ui/dialog.tsx` (existing) supports this via the `onOpenAutoFocus` prop on `DialogContent`.
**Source:** `@radix-ui/react-dialog@1.1.15` docs (verified).

### Pitfall P6: Bubble state leaking across sessions
**What goes wrong:** `/goal X` in session A, switch to session B, goal bubble from A still showing.
**Why it happens:** `useSessionStore.sessionGoals` is `Map<sessionId, string>`, but bubble state must filter by `activeSessionId` at render time.
**How to avoid:** Phase 7 D-04 already established "rendering filters by active session" (C-05). 08.2 follows the same pattern.
**Source:** Phase 7 CONTEXT.md D-04, `src/renderer/src/stores/sessionStore.ts:59` (`sessionGoals: Map<string, string>`).

### Pitfall P7: judge infinite loop
**What goes wrong:** Judge returns `satisfied: false, reason: "need more work"` indefinitely.
**Why it happens:** No max-turn limit.
**How to avoid:** Include turn cap in the goal (Claude Code pattern: `"or stop after 20 turns"`). Surface "turn N of M" in bubble (C1-04). If N reached, bubble transitions to `⏸ 暂停` and emits a sonner warning.
**Source:** Claude Code /goal docs (verified, 2026-06-05): "To bound how long a goal runs, include a turn or time clause in the condition, such as `or stop after 20 turns`."

### Pitfall P8: Frontmatter type coercion regression
**What goes wrong:** Replacing hand-rolled parser with `yaml` returns booleans as actual `true`/`false`; downstream code that does `if (frontmatter['disable-model-invocation'] === 'true')` (string compare) silently breaks.
**Why it happens:** Type change without updating consumers.
**How to avoid:** Audit all consumers of `CommandFrontmatter` before swap. Add a typed `ParsedFrontmatter` interface so consumers see `boolean | undefined` (not `string | undefined`).
**Source:** `src/main/commands/project-commands.ts:7-11` (current `CommandFrontmatter`), `src/main/commands/collectors/project.test.ts` (existing tests).

### Pitfall P9: SlashCommand IPC payload size
**What goes wrong:** Adding `bodyPath` + `frontmatter` to `SlashCommand` (shipped via `commands:list` IPC) may blow up payload when a project has 50+ custom commands.
**Why it happens:** Every IPC roundtrip serializes the full registry.
**How to avoid:** `bodyPath` is a string (1 path). `frontmatter` is a small object (~50 bytes typical). Total payload growth: 50 commands × 100 bytes = 5KB. Negligible.
**Source:** `src/main/ipc-handlers.ts:786-799` (`commands:list` handler).

### Pitfall P10: Auto-compact threshold miscalibration
**What goes wrong:** CDF uses 85% threshold; if `context_limit` is reported differently per provider (Anthropic 200k, OpenAI varies, Ollama custom), the "Free space" displayed in /context modal will be wrong.
**Why it happens:** `context_limit` is provider-specific; current `context-aggregator.ts` doesn't pass it through.
**How to avoid:** Read `context_limit` from the active provider in `llmStore` and pass to `aggregateCurrentSessionContext` as second arg. Add ASVS V5 input validation on the limit.
**Source:** `src/shared/types.ts:51` (`LLMProvider.context_limit`), `src/renderer/src/stores/llmStore.ts` (1.9KB).

### Pitfall P11: /plan modify loop creates orphan runs
**What goes wrong:** User clicks "修改" 5 times in a row → 5 LLM runs queued. Each run holds a DB row, an agent_runs entry, and memory. The 5 runs all return plans; only the last is shown.
**Why it happens:** No run cancellation when user submits a new modify opinion.
**How to avoid:** Cancel previous run via `window.electronAPI.llm.stopChat(previousRequestId)` before starting new modify run. Already exposed in `src/shared/types.ts:459`.
**Source:** `src/shared/types.ts:457-468` (`llm` namespace).

### Threat Modeling: /goal judge loop (C1-05)

- **Surface:** judge calls `runLLMChat` with goal + last N turns.
- **Trust boundary:** LLM output (JSON) is untrusted.
- **Risk:** judge could be tricked by prompt injection in the agent's last message to return `satisfied: true` prematurely.
- **Mitigation:**
  1. Parse JSON strictly; reject non-conformant output
  2. Cap turn count (P7)
  3. Log judge decisions to electron-log for audit
  4. User can `/goal clear` at any time (C-1d / C1-04)
