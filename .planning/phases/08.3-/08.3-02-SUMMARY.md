## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] parseAtTokens start offset miscalculated**
- **Found during:** Task 2 — `parseAtTokens finds a single file token`
- **Issue:** PLAN.md line 350 specified `start: 6, end: 16` for `'hello @src/foo.ts world'`. The actual `match.index` is 6 (points at `@`), so end should be `6 + 1 + 10 = 17` (path "src/foo.ts" is 10 chars, not 9). The PLAN was internally inconsistent.
- **Fix:** Updated test to `end: 17` and corrected JSDoc example. The implementation (`start: match.index`, `end: match.index + 1 + pathText.length`) is correct.
- **Files modified:** `pathUtils.test.ts`, `pathUtils.ts` (JSDoc example)
- **Commit:** `a86a5cf`

**2. [Rule 1 - Bug] normForFilter does not split precomposed U+00E9**
- **Found during:** Task 2 — `normForFilter decomposes café (U+00E9) consistently`
- **Issue:** PLAN.md test asserted `normForFilter('café') === 'cafe'`. But NFKC compatibility decomposition does NOT split precomposed U+00E9 — that is the canonical composed form. The two forms (precomposed `café` and decomposed `café`) DO normalize identically to `'café'` (the precomposed form is preserved).
- **Fix:** Updated test to assert the real contract: precomposed and decomposed forms normalize to the same string (i.e., they match each other). The function works as designed.
- **Files modified:** `pathUtils.test.ts`
- **Commit:** `a86a5cf`

**3. [Rule 1 - Bug] AtMentionPopup NFKC test used non-NFKC-decomposable character**
- **Found during:** Task 4 — `filters by NFKC-normalized substring (A-03)`
- **Issue:** Test used query `"cafe"` against candidate `"café.txt"`. Since NFKC does not split U+00E9, `'café.txt'.includes('cafe')` is false. (Same root cause as Deviation #2.)
- **Fix:** Switched to a character class that NFKC actually decomposes: the fi ligature (U+FB01 `ﬁ`). Query `"FIL"` normalizes to `"fil"` which is a substring of `"file.ts"` (the normalized ligature).
- **Files modified:** `AtMentionPopup.test.tsx`
- **Commit:** `f14dd45`

**4. [Rule 1 - Bug] cmdk mock data-testid collision with production prop**
- **Found during:** Task 4 — `renders all candidates as Command.Items`
- **Issue:** Production `<Command.Item data-testid="at-mention-item" ...>` set the same attribute that the mock's `data-testid={\`item-${value}\`}` default also set. Spread order in the mock (`{...rest}` came after `data-testid={\`item-${value}\`}`) caused the production testid to win, hiding the per-item testid from the test.
- **Fix:** Mock now strips `data-testid` from `...rest` before spreading, allowing the value-derived default to take effect.
- **Files modified:** `AtMentionPopup.test.tsx`
- **Commit:** `f14dd45`

### Plan Adjustments

**5. Test 9 parseAtTokens kind value**
- **Found during:** Task 2 — `parseAtTokens finds a single dir path (kind: file when no trailing /)`
- **Issue:** Test name in PLAN.md says "dir path" but the input is `'see @docs/intro for more'` which has no trailing `/`, so `kind: 'file'` is correct. Test name is misleading but content is correct.
- **Resolution:** Renamed test to be accurate: "(kind: file when no trailing /)".

**6. AtToken `data-at-token-ignored` attribute**
- **Found during:** Task 1 implementation
- **Issue:** PLAN said attribute should be `'true'` when ignored, omitted when not. Used conditional spread `{...(ignored ? { 'data-at-token-ignored': 'true' as const } : {})}` to keep the DOM clean (no `data-at-token-ignored="false"` noise).
