## Decisions Made

- **AtTokenSequence is local to ChatArea.tsx** (not exported, not a separate file). It's 10 lines of inline rendering — no need for a separate file. Same surgical-change principle as 08.1's `<SlashToken>` which is also co-located in ChatArea.tsx's overlay JSX.
- **AtTokenSequence uses `fontSize: 14px`** to match the composer's 14px textarea so the visual weight matches the slash token's fontSize in the composer. Welcome overlay uses 15px but AtTokenSequence is a separate visual element (not inside the inline-flex), so 14px is fine.
- **Code-block-aware at-token rendering in MessageItem**: the new `renderContentWithAtTokens` splits the text on code-block regex first, then tokenizes only the non-code segments via `renderAtSegment`. This prevents `@` inside backticks (e.g., `` `@invalid` ``) from being treated as a file path (Pitfall #7).
- **TestHarness pattern reuses the SlashCommandPopup.test.tsx approach** — re-implement the small slice of ChatArea logic in a lightweight test component rather than rendering the full ChatArea. This avoids mocking the full dependency tree (useProjectStore, useSessionStore, useLLMStore, useAgentStore, useCommandRegistry, dispatcher, etc.) while still exercising the exact production logic for the at-trigger / onSelect / Backspace code paths.
- **C-05 test uses `textarea.setSelectionRange(18, 18)`** before `fireEvent.keyDown` — the `fireEvent.keyDown` reads `currentTarget.selectionStart` from the actual DOM element, so the selection range must be set on the textarea itself.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] C-05 test `value` variable undefined in onKeyDown closure**

- **Found during:** Task 2 — `C-05: Backspace at the end of an at token atomically deletes the entire @path`
- **Issue:** The test's onKeyDown handler referenced `value` (from the onChange closure) but `value` was not defined inside the onKeyDown scope. This is a closure scoping issue: `value` is the destructured event target value, but it only exists in the onChange handler. The test was failing because `setInputVal(value.slice(...))` was receiving `undefined`.
- **Fix:** Changed `setInputVal(value.slice(0, lastAt.start))` to `setInputVal(inputVal.slice(0, lastAt.start))` — use the `inputVal` state variable that's in the TestHarness component closure.
- **Files modified:** `src/renderer/src/components/ChatArea/ChatArea.at-trigger.test.tsx`
- **Verification:** All 5 test cases pass after fix.
- **Commit:** `8cf8e65` (part of Task 2 commit)

**2. [Rule 1 - Bug] AtTokenSequence .map callback implicit any**

- **Found during:** Task 2 final tsc check
- **Issue:** TypeScript strict mode caught `(t) =>` in AtTokenSequence's `.map()` call as implicit-any. The tokens type is `ReturnType<typeof parseAtTokens>` but the parameter still needs explicit annotation in strict mode.
- **Fix:** Changed `(t) =>` to `(t: { start: number; path: string; kind: 'file' | 'dir' }) =>`
- **Files modified:** `src/renderer/src/components/ChatArea/ChatArea.tsx`
- **Verification:** `npx tsc --noEmit -p tsconfig.web.json` no longer reports this specific error.
- **Commit:** `cb926a3`
