# Phase 08.3: 增加@引用文件功能 - Research

**Researched:** 2026-06-07
**Domain:** Electron desktop app feature - chat composer @-mention file reference with `.gitignore`-aware enumeration
**Confidence:** HIGH

## Summary

Phase 08.3 在已稳定的 Phase 5/6 slash popup 框架（cmdk + Radix Popover）和 Phase 08.1 overlay inline token 模式基础上，新增 `@` 触发文件/目录引用。核心技术点：(1) 复用 `ignore@5.3.2` 解析 `.gitignore`（与 chokidar 一致），(2) 在 main 端做浅层 BFS 枚举（depth ≤ 6 + count ≤ 5000），(3) 在 renderer 端把 `@relative/path` 字面量与 08.1 inline-flex overlay 整合渲染为 `<AtToken>`，(4) IPC 通道 `project:listAtMentionCandidates(rootPath, query)` 单调用返回候选 + `truncated` 标志。整个功能不修改 dispatcher / sendMessage 契约，literal-string 协议贯穿。

**Primary recommendation:** 复制 08.1 的 overlay 模式，按 inline-flex 思路把 `@relative/path` 替换为 `<AtToken>` 渲染层；main 端新增一个 IPC handler 跑 `ignore@5` + 浅层 BFS；popup 直接复用 `cmdk + Radix Popover`。用 ZUSTAND `useAtMentionStore` 管理 popup 状态，复用 08.1 的 `useSlashCommandStore` 模板。

### Pattern 1: Inline-flex overlay 扩展（与 08.1 共存）

**What:** 在 ChatArea textarea 上方再叠一个 inline-flex 行，同时承载 slash token 和 at token。

**When to use:** 当 `parsedToken?.token`（slash）或 `parsedAtTokens`（at）非空时。

**Example:**

```tsx
// src/renderer/src/components/ChatArea/ChatArea.tsx（08.3 增量）
const parsedAtTokens = useMemo(
  () => parseAtTokens(inputVal, currentProjectRoot),
  [inputVal, currentProjectRoot]
);

<div className="flex items-start gap-1.5 w-full relative z-0" style={{ fontSize: '14px' }}>
  {parsedToken?.token && (
    <div className="flex-shrink-0 pt-[2px]">
      <SlashToken name={parsedToken.token.name} source={parsedToken.token.source} />
    </div>
  )}
  {parsedAtTokens.length > 0 && (
    <AtTokenSequence tokens={parsedAtTokens} />
  )}
  <textarea ... />
</div>
```

`<AtTokenSequence>` 自身是一个 inline-flex 容器，把 `parseAtTokens` 返回的多个 token 依次渲染。这是 KISS 思路：单组件内 inline 序列，不需要引入额外抽象。

### Pattern 2: cmdk 模糊匹配 + Radix Popover 锚定

**What:** 沿用 Phase 5/6 `SlashCommandPopup` 模板（`Command.List` + `Command.Item` + `Command.Empty`）。

**When to use:** 候选数量 0-5000，cmdk 默认 fuzzy 性能无虞。

**Example:**

```tsx
// src/renderer/src/components/AtMention/AtMentionPopup.tsx（骨架）
export const AtMentionPopup = forwardRef<AtMentionPopupHandle, AtMentionPopupProps>(
  ({ query, candidates, onSelect, onClose, loading }, ref) => {
    return (
      <Command shouldFilter label="At mention" className="w-full">
        <Command.List className="max-h-64 overflow-y-auto p-0">
          {candidates.map((c) => (
            <Command.Item
              key={c.path}
              value={c.path}  // cmdk fuzzy on path
              onSelect={() => onSelect(c)}
              className="..."
            >
              {c.kind === 'dir' ? <Folder className="w-3 h-3" /> : <File className="w-3 h-3" />}
              <span className="font-mono">{c.path}</span>
            </Command.Item>
          ))}
          <Command.Empty>未找到匹配文件</Command.Empty>
        </Command.List>
      </Command>
    );
  }
);
```

### Pattern 3: `ignore@5` 集成

**What:** 用 `ignore@5` 的 `ignore()` 工厂接受 path 模式串，返回 `ignores(path)` boolean。

**When to use:** 解析项目根 `.gitignore` + 任意嵌套 `.gitignore`（在 BFS 过程中遇到就重新实例化）。

**Example:**

```typescript
// src/main/at-mention/gitignore-loader.ts
import ignore from 'ignore';

export function loadGitignore(projectRoot: string): ReturnType<typeof ignore> {
  const ig = ignore();
  const rootGitignore = path.join(projectRoot, '.gitignore');
  if (fs.existsSync(rootGitignore)) {
    ig.add(fs.readFileSync(rootGitignore, 'utf-8'));
  }
  // B-03: 默认排除（即使未写进 .gitignore）
  ig.add(['node_modules', 'out', 'dist', '.git', '.next', '.cache']);
  return ig;
}

// 使用：ig.ignores('node_modules/foo.js') === true
// 路径必须 POSIX 风格（用 path.posix.join 拼，或 toPosix 工具）
```

### Anti-Patterns to Avoid

- **用 contentEditable 替换 textarea** — IME/选择/粘贴会回归。08.1 已锁定 overlay 方案，08.3 沿用。
- **在 renderer 端做 fs.readdir** — contextBridge 边界不允许，Node.js 集成只在 main。
- **把 @path 解析为绝对路径再传输** — 必须 relative path（CONTEXT C-04 锁定 + security：避免泄漏项目根）。
- **在 main 端做模糊匹配** — cmdk fuzzy 在 renderer 端 5000 候选无压力；main 端不必要。
- **每次按键调用 IPC** — popup 打开时一次拉 5000 候选，关闭后释放（CONTEXT E-02）。
- **atomic Backspace 用 CSS `pointer-events: none`** — 08.1 D-04 已记录，事件拦截是唯一可控方案。
- **多 token 同时渲染在 overlay** — 与 08.1 D-03 保持一致：单输入内 inline-flex 序列支持多个 at token（08.1 只支持 1 个 slash token 是产品决策，08.3 不限制 at token 数量）。

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| `.gitignore` 解析 | 自写行分割 + `picomatch` | `ignore@5` | negation `!pattern`、nested .gitignore、escaped `#`、绝对路径、空行处理全是坑 |
| 文件树 BFS | 自写队列 | `fs.readdirSync({ recursive: true, withFileTypes: true })`（Node 20+） | Node 20+ 内置 recursive 枚举，避免自维护 |
| 光标位置计算 | DOM Selection API 试探 | `textarea.selectionStart` / `selectionEnd` | 浏览器内置 API，08.1 已使用同样手法 |
| 模糊字符串匹配 | 自写 fuzzy 评分 | `cmdk` 内置 fuzzy（`shouldFilter: true`） | 08.1 已验证 5000 命令流畅；AtToken 候选 5000 同样适用 |
| NFKC + variation selector | 自写正则 | Phase 8 `SlashCommandPopup.tsx:37-40` 的 `VARIATION_SELECTORS` + `normForFilter` | 已有 + 已测试；08.3 抽出共用 util |
| popover 锚定 | 自写悬浮层 + 位置计算 | `@radix-ui/react-popover` | Radix 处理 viewport / flip / collision；08.1 沿用 |
| `getRelativePath` | 自写 | `path.relative(projectRoot, absolutePath)` + 统一 `/` 分隔 | Node 内置 |

**Key insight:** 08.3 的核心复杂度不在 UI 也不在 main 端 IO，而在**两个 token 类型（slash + at）的 overlay 同步**。08.1 已证明 overlay 模式可行；08.3 复用其模板，把新 token 类型视为同一 overlay 内的 inline-flex 子节点。**重点投资在 overlay 状态同步测试**，不要投资在 .gitignore 解析（库已处理）。

## Common Pitfalls

### Pitfall 1: `@` 触发时未处理 IME 合成
**What goes wrong:** 中文输入 `@` 时，Pinyin 候选可能临时显示在弹窗之上；用户在候选上选字时若触发 popup 打开，会被 Pinyin 界面截断。
**Why it happens:** `onChange` 触发时 `isComposingRef.current` 是 true 时未阻断 popup 打开。
**How to avoid:** 沿用 08.1 的 `isComposingKeyEvent(e)` + 200ms compositionEnd 缓冲；`setAtOpen` 调用前先检查。
**Warning signs:** 中文输入法用户报告 popup 闪烁。

### Pitfall 2: popup 关闭后未释放 5000 候选缓存
**What goes wrong:** 用户连续打开/关闭 popup 多次后 IPC 调用次数累加，内存滞留。
**Why it happens:** useState 持有 candidates 数组，popup 关闭时未清空。
**How to avoid:** `useAtMentionStore` 监听 `open=false` 主动 `setCandidates([])`；或在 `useEffect` cleanup 中清空。
**Warning signs:** DevTools Memory heap 上涨。

### Pitfall 3: BFS 递归深度未限制
**What goes wrong:** `node_modules` 嵌套层级（mono-repo）可达 50+ 层，递归栈溢出或超 5s 卡顿。
**Why it happens:** 自写 DFS 递归忘记深度计数器。
**How to avoid:** 用显式队列 BFS，传入 `depth + 1` 累计；depth > 6 跳过子目录（CONTEXT B-05）。
**Warning signs:** main 进程 100% CPU 单核；UI 冻结。

### Pitfall 4: 大仓库 5000 候选 IPC 传输延迟
**What goes wrong:** 序列化 5000 个 `{path, kind}` 对象到 IPC 边界耗时 200-500ms。
**Why it happens:** Electron IPC 用结构化克隆算法，5000 对象可超过 100KB。
**How to avoid:** (1) main 端提前按路径名排序/截断到 5000；(2) 只传 `path` 字符串数组，kind 在 renderer 端用 `path` 后缀推断（`/` → dir，否则 file）；(3) 考虑 IPC 改为 `[]string` + 单独 list 接口。本 phase 选择 **(2)**：transmit 极简 payload，renderer 端 `candidates.endsWith('/')` 推断 kind。
**Warning signs:** popup 打开时 200ms+ 延迟。

### Pitfall 5: relative path 在 Windows 上的前缀
**What goes wrong:** `path.relative('C:\\proj', 'C:\\proj\\src\\a.ts')` 返回 `'src\\a.ts'`（Windows 反斜杠），但 token 渲染和 LLM 接收都期望正斜杠。
**Why it happens:** Node.js `path.relative` 跟随平台 `path.sep`。
**How to avoid:** 工具 `getRelativePath(abs)` 内部 `path.relative().replace(/\\/g, '/')`，并在 main 端 + renderer 端共用同一工具（建议放 `src/renderer/src/lib/commands/pathUtils.ts` 并通过 IPC 复用 main 端的输出，**不要让 renderer 自己计算 relative**——CONTEXT E-01 锁定 main 端枚举）。
**Warning signs:** Windows 用户报告 token 路径含反斜杠。

### Pitfall 6: tailwind v4 静态类扫描遗漏新 token 类
**What goes wrong:** `<AtToken>` 使用了 `bg-blue-100/20` 等动态拼接类，Tailwind v4 静态扫描未发现，生产环境样式丢失。
**Why it happens:** Tailwind v4 不支持 runtime class generation（08.1 D-01..D-04 invariant 提示）。
**How to avoid:** 所有 AtToken 类必须硬编码字面量；不要用 `clsx` 拼接动态类名（用条件渲染替代）。
**Warning signs:** dev 环境正常，build 后样式丢失。

### Pitfall 7: `parseInputToTokens` 与 08.3 `parseAtTokens` 解析冲突
**What goes wrong:** 输入 `/goal @src/foo.ts` 时，slash parser 先吃掉 `/goal `，剩下 `@src/foo.ts` 给 at parser，本应正确；但若 slash parser 把整个字符串吃成 token（不识别空格后的内容），会丢失 at 解析。
**Why it happens:** 08.1 `parseInputToTokens` 解析 `/cmd-name ` 后只返回 token，不处理后续的 at tokens。
**How to avoid:** `parseAtTokens` 是独立工具，对**整个 inputVal** 做全局 `@relative/path` 扫描；slash parser 仅判断前导 token。**两个工具的输出在 overlay 渲染时合并：先 slash token（若有），再 at token 序列（若 inputVal 含 `@`）**。
**Warning signs:** `/goal @src/foo.ts` 输入后 overlay 只显示 `/goal` token，无 at token。

### Pitfall 8: `onMouseDown preventDefault` 丢失导致 token 点击夺焦
**What goes wrong:** 用户在 token 上点击，textarea 失焦，弹窗关闭，token 进入"未渲染"状态。
**Why it happens:** 08.1 C-06 防御措施；08.3 同样必须 `onMouseDown={(e) => e.preventDefault()}`。
**How to avoid:** 在 `<AtToken>` 中复制 08.1 SlashToken 的 `onMouseDown` 行为；测试覆盖。
**Warning signs:** 视觉闪烁 token 消失。

### Pitfall 9: cmdk 在 jsdom 中 portal 挂载失败
**What goes wrong:** 测试中 cmdk 的 `Command.List` 渲染时报错 "Could not find portal target"。
**Why it happens:** Radix Popover 内部用 React Portal，需要 jsdom 中 document.body 存在。
**How to avoid:** vitest setup 已用 jsdom environment（`vitest.config.ts:6`），但需确认 `@testing-library/react` 的 `render()` 完成后立即查询，不要异步等待；或在 `beforeEach` 中清理 portal target。
**Warning signs:** 测试报错 "Element type is invalid"。

### Pitfall 10: ignore@5 的路径必须 POSIX
**What goes wrong:** Windows 路径 `'src\\a.ts'` 传给 `ig.ignores()` 不匹配 `.gitignore` 中的 `src/` 模式。
**Why it happens:** ignore 库内部用 `/` 分隔符，不处理 `path.sep`。
**How to avoid:** main 端 BFS 过程中产生的路径在传入 `ig.ignores()` 前用 `path.toPosix()`（自写或 `path.replace(/\\/g, '/')`）。
**Warning signs:** Windows 仓库 .gitignore 规则不生效。

### 关键决策点

1. **同 overlay 渲染 slash + at token（推荐）** vs **两个独立 overlay 层**
   - 选 1：与 08.1 模式一致；inline-flex 一次性渲染；测试简单
   - 选 2：隔离关注点，但增加 z-index / 定位复杂度

2. **main 端传输 `path` 字符串数组 + renderer 端推断 kind（推荐）** vs **传输完整 `{path, kind}` 对象**
   - 选 1：IPC payload 小 50%；renderer 端 `path.endsWith('/')` 简单
   - 选 2：语义清晰但 IPC 边界多 50% 序列化成本

3. **popup 打开时一次拉满 5000 候选** vs **每次 query 变化拉取**
   - 选 1（CONTEXT E-02 锁定）：popup 关闭后释放，5000 候选 cmdk 模糊性能 OK
   - 选 2：每次输入都 IPC 一次，5000 候选复杂度下降但 IPC 流量大

4. **`parseInputToTokens` 改造为支持 at** vs **新增独立 `parseAtTokens`**
   - 选 2（推荐）：08.1 是 locked-decision 文件，独立工具避免回归
   - 选 1：DRY 但需要扩展 08.1 已有签名

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | no | 不涉及 |
| V3 Session Management | no | 不涉及 |
| V4 Access Control | yes | 08.3 严格限制枚举根 = `currentProject.path`；不暴露系统级路径 |
| V5 Input Validation | yes | `pathUtils.getRelativePath()` 拒绝 `..` / 绝对路径 / 非法字符；IPC handler 校验 `rootPath` 是 registered project 之一 |
| V6 Cryptography | no | 不涉及 |
| V8 File Integrity | yes | 08.3 不读文件内容（CONTEXT D-02），仅 `fs.statSync` 判断 file/dir；路径若被 .gitignore 排除则不展示 |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| 路径遍历（输入 `../../etc/passwd`） | Elevation of Privilege | `getRelativePath` 拒绝 `..` + IPC handler 校验 `rootPath` 在 DB 内 |
| 跨项目枚举（`rootPath` 指向另一项目） | Information Disclosure | IPC handler 内 `db.prepare('SELECT id FROM projects WHERE path = ?')` 校验 `rootPath` 是当前激活的 project |
| 拒绝服务（巨型仓库递归枚举卡死） | Denial of Service | B-05 depth ≤ 6 + count ≤ 5000 + IPC 超时（main 端硬编码 5s） |
| Token 注入（在历史消息中伪装 `@relative/path`） | Tampering | 历史消息渲染前过滤 — 但 08.3 不做安全过滤（CONTEXT 不要求），LLM/Agent 自行判断路径合法性 |
