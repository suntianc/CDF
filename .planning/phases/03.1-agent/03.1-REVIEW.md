## Critical Issues

### CR-01: `task` 工具输入解析失败时静默回退，导致 `goal` 为空

**File:** `src/main/llm.ts:414-421`
**Issue:**
```typescript
if (input?.task) {
  try {
    const taskPackage = JSON.parse(input.task);
    goal = taskPackage.goal || '';
  } catch {
    // ignore parse error
  }
}
```
当 `input.task` 是无效 JSON 或缺少 `goal` 字段时，错误被完全吞掉，没有任何日志记录或回退逻辑。后续会将空的 `goal` 通过 `delegated_task_start` 事件发送给前端，导致前端显示的委派任务目标为空字符串，用户无法理解该任务的目的。

**Fix:**
```typescript
if (input?.task) {
  try {
    const taskPackage = JSON.parse(input.task);
    if (taskPackage.goal) {
      goal = taskPackage.goal;
    } else {
      console.warn(`[RUNTIME] task tool missing goal field:`, input.task);
      goal = '(未提供目标)';
    }
  } catch (err) {
    console.warn(`[RUNTIME] Failed to parse task tool input:`, err);
    goal = '(任务解析失败)';
  }
}
```

### CR-02: 工具调用开始时保存消息失败静默忽略

**File:** `src/renderer/src/stores/sessionStore.ts:327-329`
**Issue:**
```typescript
window.electronAPI.db.saveMessage(prevMsg).catch((err) => {
  console.error('Failed to save intermediate assistant message:', err);
});
```
在工具执行期间，助手消息先写入 SQLite 再执行工具。如果 SQLite 保存失败（磁盘满、权限问题等），错误仅写入 console，消息丢失，用户毫不知情。流式输出继续进行，但消息未持久化，重启后用户会看到不完整的对话历史。

**Fix:**
应该在 store 层面设置 error 状态并通知 UI 层：
```typescript
window.electronAPI.db.saveMessage(prevMsg).catch((err) => {
  console.error('Failed to save intermediate assistant message:', err);
  set({ error: '消息保存失败，对话历史可能不完整' });
});
```

## Warnings

### WR-01: `agentToolCalls` 可能为空数组导致属性访问问题

**File:** `src/renderer/src/components/TaskPanel/TaskPanel.tsx:110-119`
**Issue:**
```typescript
const toolSummary = useMemo(() => {
  const total = agentToolCalls.length;
  const running = agentToolCalls.filter((toolCall) => toolCall.status === 'running').length;
  const failed = agentToolCalls.filter((toolCall) => toolCall.status === 'error');
  return { total, running, failed };
}, [agentToolCalls]);
```
在 line 213 使用 `{toolSummary.failed.length}`，在 line 217-221 使用 `toolSummary.failed.slice(0, 3).map(...)`。虽然代码逻辑上 `failed` 是数组，但如果 `agentToolCalls` 因 API 返回格式问题变成 `null` 或 `undefined` 而非空数组，会在 `.filter()` 调用时抛出 `TypeError`。

**Fix:**
添加防御性检查或依赖 Zod 等进行类型验证：
```typescript
const toolSummary = useMemo(() => {
  const calls = agentToolCalls ?? [];
  const total = calls.length;
  const running = calls.filter((toolCall) => toolCall.status === 'running').length;
  const failed = calls.filter((toolCall) => toolCall.status === 'error');
  return { total, running, failed };
}, [agentToolCalls]);
```

### WR-02: `sendMessage` 成功后未清理 `pendingToolMessages`

**File:** `src/renderer/src/stores/sessionStore.ts:203`
**Issue:**
```typescript
const pendingToolMessages = new Map<string, string[]>();
```
`pendingToolMessages` 在 Promise 内部声明，但 Promise 被赋值给 `streamPromise` 变量后从未调用 `.catch()` 或等待其完成（除了 `resolve()`/`reject()`）。如果流式响应因网络错误非正常终止，`pendingToolMessages` 会一直保留在内存中（虽然是函数局部变量，Promise GC 会回收，但风格上缺少显式清理）。

### WR-03: 数据库 migrations 中错误处理不一致

**File:** `src/main/database.ts:74-105`
**Issue:**
迁移使用了 `try/catch` 包裹 `ALTER TABLE` 语句，但部分错误（如 "duplicate column name"）被忽略，其他错误被记录到 console。这意味着迁移失败不会中断应用启动，但用户不会收到任何通知。如果生产数据库迁移失败（如磁盘满导致 ADD COLUMN 部分成功），可能造成 schema 不一致状态。

**Fix:**
```typescript
try {
  db.exec(`ALTER TABLE sessions ADD COLUMN parent_session_id TEXT REFERENCES sessions(id) ON DELETE SET NULL;`);
} catch (error: any) {
  if (!error.message.includes('duplicate column name')) {
    console.error('[DB MIGRATION] Failed to add parent_session_id:', error);
  }
}
```
对所有 migration 使用统一的错误处理策略，并考虑在开发环境抛出异常。
