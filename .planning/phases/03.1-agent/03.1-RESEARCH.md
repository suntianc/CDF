# Phase 3.1: Agent Subagent Delegation - Research

**Researched:** 2026-05-26
**Domain:** deepagents.js subagent delegation mechanism, Master Agent runtime integration
**Confidence:** HIGH (verified against deepagents v1.10.2 type definitions and source behavior)

## Summary

Phase 3.1 implements Master Agent subagent delegation using deepagents.js's built-in `task` tool and `subagents` configuration. The core mechanism involves:

1. Removing `task` from `excludedTools` in the harness profile (D-15)
2. Loading agents from `subagentIds` whitelist and mapping them to deepagents `SubAgent` specs (D-06, D-17)
3. Each subagent uses a unified `responseFormat` Zod schema for structured output (D-10)
4. Runtime wraps exceptions into the failure schema format (D-11)

**Critical gap discovered:** `Agent.slug` and `Agent.model` fields do NOT exist in the current codebase (neither in `types.ts` nor `database.ts`). These must be added during implementation. The CONTEXT.md references these fields as if they exist, but they are not yet in the code.

**Primary recommendation:** Add `slug` (UNIQUE, auto-generated from name) and `model` (nullable string) columns to the `agents` table via safe migration, and update the `Agent` TypeScript interface accordingly before building the subagent mapping logic.

### Pattern 1: Subagent Configuration from Database

**What:** Load agents from `subagentIds` whitelist and map to deepagents `SubAgent` specs.

**When to use:** When building the `subagents` array for `createDeepAgent`.

**Example:**
```typescript
// Source: runtime.ts + deepagents SubAgent interface (verified from index.d.ts)
const subagents = subagentIds?.map(id => {
  const agent = db.prepare('SELECT * FROM agents WHERE id = ?').get(id);
  const mcpServers = getAgentMcpServers(agent.id);
  const mcpRuntime = await loadMcpTools(agent.id, mcpServers);
  const { skillsSources } = resolveAgentSkillsConfig(projectPath, getAgentSkillNames(agent.id));

  return {
    name: agent.slug,           // D-03: 唯一稳定键
    description: agent.description || '',
    systemPrompt: agent.system_prompt || '',
    tools: [...mcpRuntime.tools], // D-07: 显式构造，不继承
    model: agent.model || undefined,
    responseFormat: DelegatedTaskResultSchema, // D-10
    skills: skillsSources,
  };
});
```

### Pattern 3: DelegatedTaskResultSchema (Unified Response Format)

**What:** All subagents return structured output conforming to a Zod schema.

**When to use:** D-10 mandates unified schema for all subagent responses.

**Example:**
```typescript
// Source: D-10 in CONTEXT.md
import { z } from 'zod';

export const DelegatedTaskResultSchema = z.object({
  status: z.enum(['success', 'failure']),
  artifacts: z.array(z.string()),
  summary: z.string(),
  error: z.object({
    code: z.string(),
    message: z.string(),
  }).optional(),

// Usage in subagent config:

  name: 'code-agent',
  responseFormat: DelegatedTaskResultSchema,
```

### Pattern 4: Exception Wrapping in Runtime

**What:** Runtime catches subagent exceptions and wraps them into `failure` format.

**When to use:** D-11: Runtime 统一包装异常。

**Example:**
```typescript
// Source: D-11 in CONTEXT.md
function wrapException(error: unknown): DelegatedTaskResult {
  if (error instanceof Error) {
    if (error.message.includes('timeout')) {
      return { status: 'failure', artifacts: [], summary: '', error: { code: 'TIMEOUT', message: error.message } };
    }
    if (error.message.includes('interrupt')) {
      return { status: 'failure', artifacts: [], summary: '', error: { code: 'INTERRUPTED', message: error.message } };
    }
  }
  return { status: 'failure', artifacts: [], summary: '', error: { code: 'PARSE_FAILED', message: String(error) } };
}
```

### Pattern 5: Task Tool Identification in Stream Events

**What:** Detect delegated task execution from `tool_start` events.

**When to use:** D-12: TaskPanel 根据 stream events 识别 delegated task。

**Example:**

## Common Pitfalls

### Pitfall 1: Agent.slug Not Yet in Database

**What goes wrong:** `Agent.slug` field referenced in CONTEXT.md does not exist in `types.ts` or `database.ts`. Implementation will fail to compile/run.

**Why it happens:** The CONTEXT.md was written assuming the field exists, but it was never added to the schema.

**How to avoid:** Add `slug` column to `agents` table via safe migration AND add `slug?: string` to `Agent` interface in `types.ts`. Use slugify pattern: `slug = name.toLowerCase().replace(/\s+/g, '-')`.

**Warning signs:** TypeScript error "Property 'slug' does not exist on type 'Agent'".

### Pitfall 2: Agent.model Not in Schema

**What goes wrong:** D-06 maps `model: Agent.model` but `Agent.model` does not exist in the current schema.

**Why it happens:** Same as above - fields were designed but not implemented.

**How to avoid:** Add `model` column to `agents` table (nullable TEXT) and `model?: string` to `Agent` interface.

### Pitfall 3: task Tool Parameters Mismatch

**What goes wrong:** CONTEXT.md shows `task(name, task)` but actual deepagents API is `task({ description, subagentType })`.

**Why it happens:** CONTEXT used simplified notation for readability.

**How to avoid:** Remember actual tool call format:
- `subagent_type` = `Agent.slug` (who to delegate to)
- `description` = JSON task package (what to do)

### Pitfall 4: Subagent Tools Not Explicitly Passed

**What goes wrong:** D-07 prohibits default tool inheritance, but deepagents default subagent middleware includes default tools.

**Why it happens:** Custom subagents use `defaultSubagentMiddleware` WITHOUT parent skills, but they DO get default tools unless overridden.

**How to avoid:** Always explicitly pass `tools` array in subagent config:
```typescript
{
  name: agent.slug,
  tools: [...mcpRuntime.tools, ...builtInTools], // 必须显式构造
}
```

### Pitfall 5: Missing Database Migration for New Columns

**What goes wrong:** Adding `slug` and `model` columns without safe migration will fail on existing databases.

**Why it happens:** Fresh database works, but existing database with data will error on ALTER TABLE if columns already exist (from partial migration).

**How to avoid:** Use existing migration pattern:
```typescript
try {
  db.exec(`ALTER TABLE agents ADD COLUMN slug TEXT;`);
} catch (error: any) {
  if (!error.message.includes('duplicate column name')) {
    console.error('Failed to migrate agents table (slug):', error);
  }
}
```

## Open Questions

1. **slug 生成策略**
   - What we know: `Agent.slug` is the stable key for `task(name)`, must be URL-safe
   - What's unclear: Should it be auto-generated from `name` (slugify), or user-customizable?
   - Recommendation: Auto-generated from name with uniqueness check, user can edit after creation

2. **Subagent 超时机制**
   - What we know: D-11 mentions `timeout` error code wrapping
   - What's unclear: What is the actual timeout value? Is it configurable?
   - Recommendation: Use hardcoded MVP value (e.g., 5 minutes), expose as future enhancement (Deferred Ideas)

3. **TaskPanel 委派任务节点渲染**
   - What we know: D-12 says TaskPanel shows delegated task status, D-13 defines 3 states
   - What's unclear: Should there be a separate "Delegated Tasks" section, or inline with tool calls?
   - Recommendation: Add delegated-task node type with agent name, status, and expandable result
