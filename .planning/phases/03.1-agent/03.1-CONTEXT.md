# Phase 03.1: agent - Context

**Gathered:** 2026-05-26（第二版整改）
**Status:** Ready for planning

<domain>
## Phase Boundary

实现 Master Agent 调用子Agent 的完整流程。包括：启用 deepagents 的 `task` tool、通过 `subagentIds` 白名单配置可委派 Agent、Agent 到 deepagents subagent 的稳定映射、统一 `responseFormat` 结构化输出、Runtime 负责异常状态包装、在 TaskPanel 中展示委派任务执行状态。

Phase 3.1 聚焦**子agent 调用的运行时机制**，不涉及新的 UI 组件开发（TaskPanel 已有）。

</domain>

<decisions>
## Implementation Decisions

### 1. 委派触发机制（混合模式）
- **D-01:** 默认通过 System Prompt 指令触发委派 — Master Agent 被告知「遇到 X 类型任务时使用 `task(name, task)` 委派给对应子Agent」
- **D-02:** 用户可随时在前端干预/覆盖委派决策

### 2. 子Agent 白名单配置
- **D-03:** Master Agent 通过 `subagentIds` 白名单指定允许委派的 Agent IDs（来自 agents 表）
- **D-04:** `createDeepAgentRuntime` 增加 `subagentIds?: string[]` 参数
- **D-05:** **未配置 `subagentIds` 时，`task` tool 不可用或不暴露给 Master Agent**
- **D-06:** 非白名单内的 Agent ID 无法被 `task(name)` 调用

### 3. Agent → deepagents subagent 映射
- **D-07:** 每个白名单 Agent 映射为 deepagents subagent 配置：
  ```typescript
  {
    name: string,           // Agent.slug（稳定标识符，用于 task(name) 调用）
    description?: string,   // Agent.description
    systemPrompt: string,   // Agent.system_prompt + 任务包（由 Runtime 注入）
    tools: any[],           // 从 Agent 绑定的 MCP/Skills 解析
    model?: any,           // 可选，覆盖默认模型
    responseFormat: DelegatedTaskResultSchema  // 统一 schema
  }
  ```
- **D-08:** `name` 使用 Agent 的稳定 slug/id，**不是随机 UUID**，确保 `task(name)` 调用稳定可追溯
- **D-09:** **禁止使用默认 `general-purpose` subagent** — `generalPurposeSubagent: { enabled: false }` 保持不变

### 4. 任务包（Task Payload）
- **D-10:** 子Agent **不继承父级完整 checkpoint 历史**
- **D-11:** 子Agent 只通过 `task` 参数接收任务包，包含：
  1. **当前任务目标** — 子Agent 需要完成的具体任务描述
  2. **用户约束** — 来自原始用户请求的约束条件
  3. **相关文件路径** — 任务涉及的文件路径列表
  4. **项目根目录** — 虚拟根目录路径
  5. **期望产物** — 任务完成后的预期输出/产物
  6. **必要历史摘要** — 仅与当前子任务直接相关的历史信息
- **D-12:** 任务包通过 `task(tool).arguments.task` 传递，不是修改 system prompt

### 5. 统一 responseFormat（DelegatedTaskResultSchema）
- **D-13:** 所有子Agent **强制使用统一 Zod schema**：
  ```typescript
  const DelegatedTaskResultSchema = z.object({
    status: z.enum(["success", "failure"]),
    artifacts: z.array(z.string()),
    summary: z.string(),
    error: z.object({
      code: z.string(),
      message: z.string()
    }).optional()
  });
  ```
- **D-14:** Master Agent 解析 `status`、`artifacts`、`summary`、`error` 字段决定下一步
- **D-15:** **Runtime 负责异常状态包装**：
  - `timeout` → `{ status: "failure", artifacts: [], summary: "", error: { code: "TIMEOUT", message: "..." } }`
  - `interrupted` → `{ status: "failure", artifacts: [], summary: "", error: { code: "INTERRUPTED", message: "..." } }`
  - `parse_failed` → `{ status: "failure", artifacts: [], summary: "", error: { code: "PARSE_FAILED", message: "..." } }`

### 6. 执行可视化（TaskPanel）
- **D-16:** TaskPanel 根据 **stream metadata 和 task events** 识别 delegated task
- **D-17:** 展示内容：agentName、task 描述、执行状态、流式输出片段、最终结果
- **D-18:** 执行状态映射：`running` | `success` | `failure`（包含 timeout/interrupted/parse_failed）
- **D-19:** TaskPanel 已存在，只需扩展支持 `delegated-task` 节点类型

### 7. 底层机制变更
- **D-20:** 移除 `excludedTools: ['task']` — 启用 deepagents 的 task tool
- **D-21:** `generalPurposeSubagent: { enabled: false }` 保持不变
- **D-22:** `createDeepAgentRuntime` 接收 `subagentIds`，从 agents 表加载并映射为 `subagents[]`
- **D-23:** `task(name, task)` 调用格式：`name` 为 Agent slug，`task` 为任务包 JSON 字符串

</decisions>

<specifics>
## Specific Ideas

- `task` 参数传递 JSON 字符串化的任务包，Runtime 负责解析和注入
- Agent 的 `slug` 字段作为稳定的 `task(name)` 标识符
- 统一 `DelegatedTaskResultSchema` 确保所有子Agent 输出格式一致
- Runtime 在捕获异常时主动包装为符合 schema 的失败响应

</specifics>
