# Phase 03.1: agent - Context

**Gathered:** 2026-05-26
**Status:** Ready for planning

<domain>
## Phase Boundary

实现 Master Agent 调用子Agent 的完整流程。包括：启用 deepagents 的 `task` tool、配置 subagents 列表、实现上下文传递机制、结果汇报结构化格式、在 TaskPanel 中展示委派任务执行状态。

Phase 3.1 聚焦**子agent 调用的运行时机制**，不涉及新的 UI 组件开发（TaskPanel 已有）。

</domain>

<decisions>
## Implementation Decisions

### 1. 委派触发机制（混合模式）
- **D-01:** 默认通过 System Prompt 指令触发委派 — Master Agent 被告知「遇到 X 类型任务时使用 task tool 委派给对应子Agent」
- **D-02:** 用户可随时在前端干预/覆盖委派决策 — Master Agent 的 system prompt 中保留「可被用户干预」的指令

### 2. 子Agent 配置（Agent 资产库）
- **D-03:** 子Agent 直接复用 Agent 资产库中已定义的 Agent — 无需独立的 subagent 注册表
- **D-04:** Master Agent 通过 `subagentIds` 配置项指定允许委派哪些子Agent（来自 agents 表）
- **D-05:** `createDeepAgentRuntime` 需要增加 `subagentIds?: string[]` 参数，从 agents 表加载对应的 Agent 配置

### 3. 上下文传递（Master 负责全局状态）
- **D-06:** Master Agent 保持全局任务状态，负责分解任务和汇总结果
- **D-07:** 子Agent 只接收完成当前子任务所需的最小上下文 — 当前消息 + 项目根目录，不继承父级完整检查点历史
- **D-08:** 子Agent 的 system prompt 中包含任务描述（来自 Master 的 task tool call）

### 4. 结果汇报（结构化 JSON）
- **D-09:** 子Agent 必须返回结构化 JSON，包含：`status`（success/failure）、`artifacts`（产物路径列表）、`summary`（关键信息摘要）
- **D-10:** Master Agent 解析结构化结果后决定下一步操作
- **D-11:** 子Agent 的 system prompt 中明确定义输出 schema，确保 JSON 格式稳定

### 5. 执行可视化（TaskPanel 实时状态）
- **D-12:** 委派任务在 TaskPanel 中实时展示：子Agent 名称、执行状态、流式输出片段
- **D-13:** TaskPanel 已存在，只需扩展支持显示「委派任务」类型的节点

### 6. 底层机制变更
- **D-14:** 移除 `excludedTools: ['task']` — 启用 deepagents 的 task tool
- **D-15:** 将 `registerHarnessProfile` 中的 `generalPurposeSubagent: { enabled: false }` 改为 `true`
- **D-16:** `createDeepAgentRuntime` 需要接收 `subagentIds` 并构造 `subagents` 配置对象传给 `createDeepAgent()`

</decisions>

<specifics>
## Specific Ideas

- 子Agent 的 system prompt 由 Master Agent 在发起 task tool call 时动态注入
- 最小上下文 = 当前消息内容 + 项目根目录路径 + 子任务描述
- 结构化 JSON 结果通过 deepagents 的流式机制返回给 Master Agent

</specifics>
