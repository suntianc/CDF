<decisions>
## Implementation Decisions

### 1. 委派触发机制（混合模式）
- **D-01:** 默认通过 System Prompt 指令触发委派 — Master Agent 被告知「遇到 X 类型任务时使用 `task(name, task)` 委派给对应子Agent」
- **D-02:** 用户可随时在前端干预/覆盖委派决策 — Master Agent 的 system prompt 中保留「可被用户干预」的指令

### 2. 子Agent 配置（Agent 资产库 + 显式映射）
- **D-03:** 子Agent 直接复用 Agent 资产库中已定义的 Agent — 无需独立的 subagent 注册表
- **D-04:** Master Agent 通过 `subagentIds` 配置项指定允许委派哪些子Agent（来自 agents 表）
- **D-05:** `createDeepAgentRuntime` 需要增加 `subagentIds?: string[]` 参数
- **D-06:** Agent 映射为 deepagents subagents 时，结构为：
  ```typescript
  {
    name: string,           // Agent.name — task(name) 调用时使用
    description?: string,   // Agent.description
    systemPrompt: string,   // Agent.system_prompt
    tools: any[],           // 从 Agent 绑定的 MCP/Skills 解析
    model?: any,            // 可选，覆盖默认模型
    responseFormat?: object // Zod schema，结构化输出
  }
  ```
- **D-07:** **禁止使用默认 `general-purpose` subagent** — `registerHarnessProfile` 时显式覆盖 `generalPurposeSubagent: { enabled: false }`；或若需启用，必须将 `general-purpose` 也映射为可控的资产库 Agent

### 3. 任务包上下文（最小上下文）
- **D-08:** Master Agent 保持全局任务状态，负责分解任务和汇总结果
- **D-09:** 子Agent **不继承父级完整 checkpoint 历史**，只接收任务包（Task Payload），包含：
  1. **当前任务目标** — 子Agent 需要完成的具体任务描述
  2. **用户约束** — 来自原始用户请求的约束条件
  3. **相关文件路径** — 任务涉及的文件路径列表（避免子Agent 自行探索）
  4. **项目根目录** — 虚拟根目录路径
  5. **期望产物** — 任务完成后的预期输出/产物
  6. **输出 schema** — 期望的结构化输出格式（用于 responseFormat）
  7. **必要历史摘要** — 仅与当前子任务直接相关的历史信息（如前序步骤的产物）

### 4. 结果汇报（responseFormat 结构化输出）
- **D-10:** 子Agent **必须使用 `responseFormat` / Zod schema** 定义输出格式，不依赖 system prompt 约束 JSON
- **D-11:** 结构化输出 schema：
  ```typescript
  {
    status: "success" | "failure" | "timeout",
    artifacts: string[],           // 产物路径列表
    summary: string,              // 关键信息摘要
    error?: {                     // 失败时填写
      code: string,
      message: string
    }
  }
  ```
- **D-12:** Master Agent 解析结构化结果后决定下一步操作
- **D-13:** **JSON 解析失败 fallback** — 若子Agent 返回无法解析的输出，Master Agent 将其作为普通消息处理，并记录 `parse_failed` 状态

### 5. 执行可视化（TaskPanel 实时状态）
- **D-14:** TaskPanel 根据 stream metadata / task tool event 识别 delegated task
- **D-15:** 展示内容：agentName、执行状态（running/success/failure/timeout/parse_failed）、流式输出片段、最终结果
- **D-16:** TaskPanel 已存在，只需扩展支持显示「委派任务」类型的节点

### 6. 基础错误处理（MVP 必须）
- **D-17:** **子Agent 失败状态** — `status: "failure"` 时 TaskPanel 显示 error message
- **D-18:** **超时保护** — 子Agent 执行设置超时阈值（可配置，默认 5 分钟），超时时 `status: "timeout"`
- **D-19:** **用户取消/中断** — Master Agent 收到中断信号后，将子Agent 状态回写为 `interrupted`，TaskPanel 同步展示
- **D-20:** **状态回写** — 委派任务的最终状态（success/failure/timeout/interrupted）需要回写到 Master Agent 的 checkpoint，确保重启后可恢复

### 7. 底层机制变更
- **D-21:** 移除 `excludedTools: ['task']` — 启用 deepagents 的 task tool
- **D-22:** `registerHarnessProfile` 中 `generalPurposeSubagent: { enabled: false }` 保持不变，不启用默认 general-purpose
- **D-23:** `createDeepAgentRuntime` 接收 `subagentIds` 并从 agents 表加载，映射为 `subagents` 配置对象传给 `createDeepAgent()`
- **D-24:** `task` tool 调用格式为 `task(name="子Agent名称", task="任务描述")`

</decisions>

<specifics>
## Specific Ideas

- 子Agent 的 system prompt = Agent 资产库中的 system_prompt + 任务包内容（由 Master Agent 在发起 task call 时注入）
- 任务包通过 `task` tool 的 `task` 参数传递，不是通过修改 system prompt
- `responseFormat` 使用 Zod schema 确保 JSON 输出稳定性

</specifics>
