## 1. System Classification

**System Type:** Multi-Agent (Subagent Delegation)

**Description:**
Master Agent delegates sub-tasks to specialized subagents via deepagents.js `task(name, task)` tool. The Master Agent maintains global task state, decomposes work, and coordinates subagent execution. Subagents receive minimal context (task package via `task` parameter) and return structured JSON results. The system enforces a strict whitelist (`subagentIds`) controlling which agents can be delegated to. Runtime wraps exceptions (timeout, interrupted, parse_failed) into standardized failure responses.

**Critical Failure Modes:**
1. Subagent called with wrong `name` — Master sends `task(name="invalid-agent", ...)` and delegation fails silently or routes to wrong agent
2. Subagent returns non-JSON or schema-violating output — Master cannot parse result, `parse_failed` status not properly propagated
3. Master does not detect subagent failure — task appears `running` in TaskPanel indefinitely (timeout never fires)
4. Unauthorized subagent called — `task` tool exposes agents not in `subagentIds` whitelist
5. Task package missing critical context — subagent cannot determine task boundaries, produces incorrect artifacts

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer Tooling (Agent Development Workstation)

**User Population:** Developers using natural language to drive automated development workflows via Master Agent; subagents handle specialized tasks (code generation, review, testing, documentation)

**Stakes Level:** Medium — incorrect subagent artifacts may require manual review before use

**Output Consequence:** Subagent artifacts become project files; incorrect outputs may introduce bugs if merged without review

### What Domain Experts Evaluate Against

```
Dimension: Delegation scope correctness
Good: Master Agent sends `task(name)` where `name` exactly matches an Agent.slug from the subagentIds whitelist. The subagent invoked matches the task's specialized domain (e.g., code-agent for implementation, review-agent for code review).
Bad: Master delegates to wrong subagent due to ambiguous task description, OR Master attempts the work itself instead of delegating to a whitelisted subagent.
Stakes: High
Source: deepagents.js documentation — "Subagent not being called" and "Wrong subagent selected" are documented failure modes; D-03 (slug as stable key) and D-05 (general-purpose disabled) are explicit design constraints
```

```
Dimension: Structured output compliance
Good: Subagent returns valid JSON conforming to `DelegatedTaskResultSchema` — `status` is "success" or "failure", `artifacts` is an array of file paths/strings, `summary` is a plain-text description. On failure, `error.code` and `error.message` are populated.
Bad: Subagent returns free-text narrative, Markdown code blocks, or JSON violating the schema (missing required fields, wrong types). Parser `safeParse()` fails and Runtime must wrap as `PARSE_FAILED`.
Stakes: High
Source: D-10 (responseFormat schema enforcement) and D-11 (Runtime exception wrapping); deepagents docs on structured output requirement
```

```
Dimension: Error state propagation
Good: When a subagent times out, is interrupted, or returns unparseable output, the TaskPanel shows `failure` status with the correct `error.code` (TIMEOUT, INTERRUPTED, PARSE_FAILED). The Master Agent can detect and handle the failure.
Bad: TaskPanel shows `running` indefinitely after a subagent fails; error is silently swallowed or logged without surfacing to UI. User has no indication the delegation failed.
Stakes: High
Source: D-11/D-14 (Runtime exception wrapping); deepagents "Subagent not being called" failure mode — tasks that should fail silently appear to continue
```

```
Dimension: Task package completeness
Good: Task package includes all six fields: `goal` (what to accomplish), `constraints` (user-specified limits), `files` (relevant file paths), `rootDir` (project root), `expected` (deliverable description), `history` (relevant conversation context). Subagent has everything needed to determine task boundaries.
Bad: Task package missing files, history, or constraints — subagent produces artifacts outside the user's stated requirements or ignores project conventions. Developer must redo work.
Stakes: Medium
Source: D-09 (task package fields); deepagents docs on context quarantine vs context propagation tradeoff
```

```
Dimension: Artifact quality
Good: Subagent produces file paths that exist in the project, code that matches project conventions, and artifacts that directly address the `goal`. Summary accurately describes what was produced.
Bad: Subagent produces non-existent file paths, code with syntax errors or wrong imports, or artifacts unrelated to the stated goal. Artifacts require significant rework before being usable.
Stakes: Medium
Source: Developer workflow practitioner knowledge — artifact quality directly affects downstream development velocity; incorrect outputs introduce bugs if merged without review
```

### Known Failure Modes in This Domain

1. **Wrong subagent selected due to ambiguous scope**: Master Agent has insufficient differentiation between subagent descriptions. A task about "optimizing this query" routes to the documentation agent instead of the data-agent. Fix: explicit scope descriptions in Agent definitions.

2. **Context bloat from unfiltered subagent returns**: Subagent returns raw file contents, full test output, or verbose debugging logs instead of summaries. Master Agent context window fills with raw data. Fix: instruct subagents to return summaries only; save large data to filesystem (deepagents docs).

3. **Silent delegation failure**: Master calls `task(name, task)` where `name` is not in `subagentIds` whitelist. deepagents throws a runtime error that is caught but not surfaced to TaskPanel. Task appears to succeed but no work was done. Fix: pre-validation guardrail checking slug against whitelist before invocation.

4. **General-purpose agent leakage**: If `generalPurposeSubagent: { enabled: true }` or the harness profile key does not match, the built-in general agent becomes available via `task` tool. Master may delegate to it instead of whitelisted agents, bypassing tool restrictions and skill bindings. Fix: D-05 explicitly disables general-purpose; verify profile key matches model name exactly (deepagents docs pitfall).

## 2. Framework Decision

**Selected Framework:** deepagents.js

**Version:** ^1.10.2

**Rationale:**
Phase 3.1 implements Master Agent subagent delegation within the existing deepagents.js runtime. The context decisions (D-01 through D-18) explicitly commit to deepagents.js as the underlying framework. deepagents.js provides:
- Built-in `task` tool for subagent delegation
- `subagents` configuration with name, description, systemPrompt, tools, model, responseFormat
- Native checkpointing and state persistence
- LangChain ecosystem integration (compatible with project's @langchain/* dependencies)

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| CrewAI | Python-only, not TypeScript; poor checkpointing for stateful workflows |
| LangGraph | Would require replacing deepagents.js; adds unnecessary complexity |
| OpenAI Agents SDK | Vendor lock-in to OpenAI; project requires model-agnostic |

**Vendor Lock-In Accepted:** Partial — deepagents.js is LangChain's agent harness, creating moderate coupling to LangChain ecosystem. Acceptable because: (1) already selected in Phase 3, (2) provides exactly the subagent primitives needed, (3) project already depends on @langchain/* packages.

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `SubAgent` config object | `{ name, description, systemPrompt, tools, model, responseFormat }` — one entry per whitelisted Agent | D-06: 将每个 `subagentIds` 条目映射为 subagent |
| `task(name, task)` tool | Master Agent 内置工具，调用时传入 `name=Agent.slug` 和 `task=JSON.stringify(taskPackage)` | D-18: 委派任务给子Agent |
| `responseFormat` | Zod schema 或 JSON schema，强制子Agent 输出结构化结果 | D-10: 统一所有子Agent 返回 `DelegatedTaskResultSchema` |
| `registerHarnessProfile` | 注册 agent model 配置（generalPurposeSubagent、excludedTools） | D-05/D-15/D-16: 禁用 general-purpose、启用 task tool |
| `interruptOn` | 人机交互配置，特定工具触发审批流程 | 配置写文件/编辑文件/删除文件的审批 |
| `generalPurposeSubagent: { enabled: false }` | 禁用内置通用子Agent | D-05: 默认禁用，不接受任何自动启用 |

### Common Pitfalls

1. **`excludedTools: ['task']` 导致 task tool 不暴露** — runtime.ts 第 215 行将 task 排除在外的黑名单配置，Master Agent 无法看到 task 工具，导致所有委派调用静默失败。**修复：设为 `[]` 或不传 `excludedTools`**。

2. **`task` 调用参数类型错误 — 传入对象而非字符串** — deepagents `task` 工具的 `task` 参数必须是字符串（任务描述 JSON 字符串）。传入对象会导致工具调用失败。**必须使用 `JSON.stringify(taskPackage)`**。

3. **子Agent 不继承父级 checkpoint（设计决策，非 bug）** — D-08 明确要求子Agent 不继承父级 checkpoint，任务包只能通过 `task` 参数传递。如果期望共享状态，必须在任务包中包含相关上下文。

4. **generalPurposeSubagent 未正确禁用** — `registerHarnessProfile` 的 key 必须与 `createDeepAgent` 的 `model` 参数完全匹配（如 `"anthropic:claude-sonnet-4-6"`）。如果 key 不匹配，禁用规则不会生效，仍可能启用通用子Agent。

5. **responseFormat 与子Agent systemPrompt 不一致** — 如果子Agent 的 systemPrompt 没有明确要求返回符合 schema 的 JSON，子Agent 可能返回自由文本导致解析失败。systemPrompt 必须包含 `responseFormat` 的约束说明（如 "你必须返回符合以下 schema 的 JSON..."）。

**Runtime 异常包装（retry 和 parse_failed 处理）：**

```typescript
async function executeSubagentTask(
  deepAgent: ReturnType<typeof createDeepAgent>,
  taskPackage: DelegatedTaskPackage,
  maxRetries = 2
): Promise<DelegatedTaskResult> {
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const result = await deepAgent.invoke({ messages: [{ role: 'user', content: JSON.stringify(taskPackage) }] });
      const parsed = DelegatedTaskResultSchema.safeParse(result);
      if (parsed.success) return parsed.data;
      // Schema 验证失败
      console.error(`[SUBAGENT] Parse failed (attempt ${attempt + 1}):`, parsed.error);
    } catch (error: any) {
      console.error(`[SUBAGENT] Execution error (attempt ${attempt + 1}):`, error.message);
      // D-11: 包装异常为 failure 格式
      if (error.message?.includes('timeout') || error.message?.includes('TimedOut')) {
        return { status: 'failure', artifacts: [], summary: '', error: { code: 'TIMEOUT', message: error.message } };
      }
      if (error.message?.includes('interrupted') || error.message?.includes('Cancelled')) {
        return { status: 'failure', artifacts: [], summary: '', error: { code: 'INTERRUPTED', message: error.message } };
      }
    }
  }
  // 所有重试失败
  return { status: 'failure', artifacts: [], summary: '', error: { code: 'PARSE_FAILED', message: 'Max retries exceeded' } };
}
```

### 4b.2 Async-First Design

**deepagents.js 中的 async 处理：**

- `createDeepAgent` 返回的 agent 实例方法（`invoke`, `streamEvents`）均为 async
- `streamEvents` 用于流式输出（TaskPanel 实时展示）
- `invoke` 用于等待完整结果（结构化输出验证）

**常见错误：在已运行的 event loop 中调用 `asyncio.run()`**

deepagents.js (TypeScript/Javascript) 不存在此问题，但有一个等效陷阱：**在 `streamEvents` 循环内尝试 `await` 一个已经 stream 的结果**。正确做法是用 `for await...of` 迭代流事件，或使用 `invoke` 等待完整结果。

**Stream vs. Await 选择：**

| 场景 | 方式 | 原因 |
|------|------|------|
| TaskPanel 实时展示委派进度 | `streamEvents` | 用户需要看到中间输出 |
| 验证结构化输出 | `invoke` + `DelegatedTaskResultSchema.parse()` | 确保完整结果一次性验证 |
| 超时控制 | `Promise.race([invoke, timeout()])` | 外层超时包装 |

```typescript
// TaskPanel 流式集成
async function* streamSubagentTask(agent: any, taskPackage: DelegatedTaskPackage) {
  const stream = await agent.streamEvents({
    messages: [{ role: 'user', content: JSON.stringify(taskPackage) }],
  });
  for await (const event of stream) {
    if (event.event === 'on_tool_start') {
      yield { type: 'tool_call', name: event.name, input: event.input };
    }
    if (event.event === 'on_tool_end') {
      yield { type: 'tool_result', output: event.output };
    }
    if (event.event === 'on_agent_finish') {
      yield { type: 'finish', output: event.output };
    }
  }
}
```

| 子Agent 委派（Sonnet） | claude-sonnet-4-6 | 1500 tokens | 800 tokens | ~$0.0012 |

**成本优化策略：**

1. **子Agent 使用更便宜的模型** — 大多数专项任务（代码搜索、格式化、简单审查）使用 Haiku 级别即可
2. **exact-match 缓存** — 对完全相同的 `taskPackage.goal + constraints` 组合缓存结果，命中则跳过委派
3. **Semantic 缓存** — 对相似任务（embedding cosine similarity > 0.95）返回缓存结果
4. **降级策略** — 如果子Agent 连续失败 2 次，自动降级到 Master Agent 直接处理

}

const DEFAULT_COST_BUDGET: CostBudget = {

## 5. Evaluation Strategy

### 5.1 Eval Dimensions and Rubrics

All dimensions are grounded in domain rubric ingredients from Section 1b. Each rubric defines domain-specific pass/fail behavior in developer-tooling language, not generic labels.

---

#### Dimension 1: Delegation Scope Correctness

> Domain source: Section 1b "Delegation scope correctness"

**Rubric:**

> **PASS:** Master Agent calls `task(name)` where `name` exactly matches an `Agent.slug` present in the `subagentIds` whitelist. The `task` tool is visible to the Master Agent (i.e., `excludedTools` does not contain `'task'`). No call is made to any agent outside `subagentIds`. The `generalPurposeSubagent` profile is disabled.

> **FAIL:** Master Agent calls `task` with a slug not in `subagentIds`; OR `task` tool is not visible because `excludedTools` contains `'task'`; OR the general-purpose agent is available via `task` tool due to a mismatched profile key.

**Measurement:** Code (unit test + integration test)
- Unit test: Mock `createDeepAgentRuntime` with known `subagentIds`. Capture `task` tool calls. Assert `name` argument is in the whitelist.
- Integration test: Invoke full runtime with a Master Agent prompt designed to trigger delegation. Verify tool call `name` matches a whitelisted slug.
- Regression test: Verify `registerHarnessProfile` is called with `excludedTools: []` and `generalPurposeSubagent: { enabled: false }`.

**Priority:** Critical

---

#### Dimension 2: Structured Output Compliance

> Domain source: Section 1b "Structured output compliance"

**Rubric:**

> **PASS:** Subagent output is valid JSON that passes `DelegatedTaskResultSchema.safeParse()`. On `status: "success"`, `artifacts` is a non-null array of strings, `summary` is a string. On `status: "failure"`, `error.code` and `error.message` are populated strings. No Markdown, no trailing text outside the JSON object.

> **FAIL:** Output is a raw string that is not JSON; OR JSON but missing required fields (`status`, `artifacts`, `summary`); OR `artifacts` is not an array; OR `error` object missing required `code`/`message` on failure; OR free-text narrative or Markdown code blocks surround the JSON.

**Measurement:** Code (integration test)
- Mock subagent returning a `DelegatedTaskResult`-shaped object. Test `safeParse` with valid output, with missing fields, with wrong types, with Markdown wrapper. Assert pass/fail as expected.
- End-to-end integration: Call subagent with real model (or mock adapter) and verify raw output from `agent.invoke()` passes schema.

**Priority:** Critical

---

#### Dimension 3: Error State Propagation

> Domain source: Section 1b "Error state propagation"

**Rubric:**

> **PASS:** When subagent execution encounters a timeout, interrupt signal, or parse failure, the TaskPanel receives a `DelegatedTaskResult` with `status: "failure"` and the correct `error.code` (`TIMEOUT`, `INTERRUPTED`, or `PARSE_FAILED`). The task does not remain in `running` state. The error is logged with full context.

> **FAIL:** Task stays in `running` state after subagent exits; OR error is silently swallowed (caught but not returned to caller); OR wrong `error.code` is set (e.g., `TIMEOUT` for a parse failure); OR error logged only to console, not propagated to TaskPanel.

**Measurement:** Code (integration test) + Human (calibration)
- Unit: Mock deepagents `invoke` to throw `Error('TimedOut')`, `Error('Cancelled')`, and to return non-JSON string. Assert each error type maps to the correct `error.code`.
- Integration: With a real model (or mocked adapter), trigger each error path via appropriate inputs. Verify TaskPanel event payload.
- Human calibration: Review 3-5 actual timeout/interrupt traces to confirm error codes are semantically correct.

**Priority:** Critical

---

#### Dimension 4: Task Package Completeness

> Domain source: Section 1b "Task package completeness"

**Rubric:**

> **PASS:** `buildTaskPackage()` returns an object with all 6 fields populated: `goal` (non-empty string), `constraints` (array, may be empty), `files` (array of path strings), `rootDir` (string), `expected` (non-empty string), `history` (string, truncated if necessary). No field is `undefined` or `null`.

> **FAIL:** Any required field is missing or undefined; OR `goal`/`expected` are empty strings (subagent has no direction); OR `files` is not an array; OR `rootDir` is not a string.

**Measurement:** Code (unit test)
- Call `buildTaskPackage()` with various inputs including missing optional fields. Assert all 6 fields are present and typed correctly.
- Test truncation: Provide `history` exceeding `MAX_HISTORY_CHARS` (8000). Assert truncated result.

**Priority:** High

---

#### Dimension 5: Artifact Quality

> Domain source: Section 1b "Artifact quality"

**Rubric:**

> **PASS:** Subagent returns `artifacts` containing file paths that exist under `rootDir` within the virtual FilesystemBackend. Code artifacts parse without syntax errors (if applicable). `summary` accurately describes the produced artifacts. No artifacts are generated that are entirely unrelated to the `goal`.

> **FAIL:** `artifacts` contains paths that do not exist in the virtual filesystem; OR code artifacts fail basic syntax validation; OR `summary` describes work that does not match `artifacts`; OR artifacts clearly contradict the `goal` (e.g., "implement login" requested but documentation generated).

**Measurement:** LLM Judge + Human (spot-check)
- LLM Judge: Prompt a calibrated evaluator model: "Given task goal '[goal]' and artifacts '[artifacts]', does the subagent output directly address the goal? Rate 1-5." Use threshold >= 3 as pass.
- Human spot-check: Review 10% of delegation artifacts for file path validity and convention compliance. Senior developer labels pass/fail.
- Code: For code artifacts, run `tsc --noEmit` or `eslint` on the generated content via virtual filesystem.

**Priority:** Medium

---

#### Dimension 6: Context Window Safety

> Domain source: Section 1b and D-09 task package fields; Section 4b.4 context window management

**Rubric:**

> **PASS:** `buildTaskPackage()` truncates `history` to `MAX_HISTORY_CHARS` (8000) when exceeded. The resulting JSON string length of the task package does not exceed `MAX_TASK_PACKAGE_CHARS` (10000). No data loss except intentional truncation of oldest history entries.

> **FAIL:** `history` is passed through untruncated when it exceeds the limit; OR the total task package JSON exceeds 10000 characters without truncation or warning; OR truncation removes the most recent history instead of the oldest.

**Measurement:** Code (unit test)
- Test with `history` at exactly 7999, 8000, and 12000 chars. Assert truncation behavior.
- Test with full task package (all fields maxed). Assert total JSON string length <= 10000.

**Priority:** Medium

---

#### Dimension 7: Unauthorized Delegation Prevention

> Domain source: Critical failure mode #4; Section 1b "Known Failure Mode: Silent delegation failure"

**Rubric:**

> **PASS:** Before invoking `task(name, task)`, the runtime validates that `name` is in the `subagentIds` whitelist. If not, the delegation is blocked and an error is returned to the Master Agent immediately — not passed to deepagents where it would raise an uncaptured exception.

> **FAIL:** `task(name, task)` is called with a slug not in `subagentIds`, allowing deepagents to either fail silently or route to an unintended agent. No pre-validation guard is in place.

**Measurement:** Code (unit test)
- Call the guard function with a slug in the whitelist → assert no error.
- Call with a slug not in the whitelist → assert error thrown (`DelegationDeniedError`) with `code: 'UNAUTHORIZED_AGENT'`.
- Integration: Attempt delegation from Master Agent with a non-whitelisted agent name. Assert TaskPanel shows `failure` with `UNAUTHORIZED_AGENT`, not `running` forever.

**Priority:** Critical

## 6. Guardrails

### 6.1 Online Guardrails (Real-Time, Catastrophic)

Each runs on every request. Latency impact must be < 5ms.

| Guardrail | Trigger Condition | Intervention | Latency |
|-----------|------------------|--------------|---------|
| **Whitelist pre-validation** | `task(name, task)` called — before passing to deepagents | If `name` not in `subagentIds`, throw `DelegationDeniedError` with `code: 'UNAUTHORIZED_AGENT'`. Do not call deepagents. | < 1ms |
| **Task package size guard** | `JSON.stringify(taskPackage).length > MAX_TASK_PACKAGE_CHARS` (10000) | Truncate `history` to fit. Log `WARN: task_package_truncated`. If still > 10000 after truncation, block with `DelegationDeniedError` code `PACKAGE_TOO_LARGE`. | < 1ms |
| **Schema validation on subagent result** | Subagent `invoke()` returns — before passing to Master | If `DelegatedTaskResultSchema.safeParse()` fails, wrap as `{ status: "failure", error: { code: "PARSE_FAILED", message: "..." } }`. Do not crash Master. Log raw output. | < 2ms |
| **Timeout enforcement** | `Promise.race([subagent.invoke(), timeout(MAX_SUBAGENT_CALL_MS)])` | If timeout wins, cancel subagent invoke and return `{ status: "failure", error: { code: "TIMEOUT", message: "..." } }`. | < 1ms |

### 6.2 Offline Flywheel (Quality Signal, Batch Analysis)

Sampled batch analysis feeding the improvement loop. No latency impact on request path.

| Metric | What It Tracks | Sampling | Degradation Action |
|--------|---------------|----------|-------------------|
| **Delegation success rate** | `task()` calls returning `status: "success"` / total calls | 100% logging; 5% random sample for human review | Below 90%: alert + freeze new sessions |
| **Parse failure rate** | `PARSE_FAILED` errors / total subagent returns | 100% logging; 10% sample reviewed for root cause | Below 95% parse success: review subagent system prompts |
| **Wrong-subagent rate** | `task(name)` where `name` != intended domain (inferred from goal text) | 20% sample reviewed by senior developer | Above 5%: improve Agent descriptions for differentiation |
| **Cost per session** | Cumulative estimated cost vs `DEFAULT_COST_BUDGET.maxCostPerTask` ($0.05) | 100% tracking; alert at 2x budget | Exceed 2x: disable new subagent calls for session |
| **TaskPanel state correctness** | Tasks reaching terminal `success`/`failure` state / total tasks | 10% sample reviewed for correct state transitions | Stuck `running`: investigate timeout enforcement |

## 7. Production Monitoring

### 7.1 Tracing Tool

**Selected:** Arize Phoenix (open-source, self-hostable, LangChain-instrumentable)

- deepagents.js is LangChain-based; `LangChainInstrumentor()` from `@langchain/langchain-core/opentelemetry` instruments all LangChain calls (including deepagents internals)
- Phoenix receives traces via OpenTelemetry Protocol (OTLP) — `http://localhost:6006/v1/traces` by default
- Self-hostable: run `px.launch_app()` locally or deploy Phoenix server
- Alternative: Langfuse (if team prefers hosted). Both are supported via LangChain callback interface.

### 7.2 Key Metrics

| Metric | Type | Description |
|--------|------|-------------|
| `subagent.delegation.count` | Counter | Total `task(name, task)` calls per session |
| `subagent.delegation.success_count` | Counter | Calls returning `status: "success"` |
| `subagent.delegation.failure_count` | Counter | Calls returning `status: "failure"` by `error.code` |
| `subagent.delegation.success_rate` | Gauge | `success_count / delegation_count` (rolling 5-min window) |
| `subagent.delegation.latency_ms` | Histogram | `invoke()` duration per subagent model (p50, p95, p99) |
| `subagent.delegation.cost_estimate` | Counter | Cumulative cost per session (estimated from token counts) |
| `subagent.task.package_size` | Histogram | `JSON.stringify(taskPackage).length` distribution |
| `subagent.task.truncation_count` | Counter | Tasks where history was truncated |
| `subagent.guardrail.unauthorized_blocked` | Counter | Delegations blocked by whitelist pre-validation |
| `taskpanel.state.running_stuck` | Counter | Tasks stuck in `running` > 5 minutes without transition |

### 7.3 Alert Thresholds

| Alert | Condition | Severity | Action |
|-------|-----------|----------|--------|
| **Delegation success rate drop** | `success_rate < 0.90` sustained 5 min | Page | Freeze new sessions; page on-call |
| **Parse failure spike** | `PARSE_FAILED rate > 5%` in any 1-min window | Page | Review subagent system prompts; check model API |
| **Timeout storm** | `TIMEOUT` errors > 10 in 5 min | Page | Check model API health; increase timeout threshold |
| **Latency degradation** | `latency_p95 > 60000ms` | Warn | Log; page if sustained > 3 min |
| **Cost overrun** | `cost_estimate > $0.50` per task | Warn | Alert; page if > $1.00 |
| **Stuck running tasks** | Any task in `running` > 10 min without transition | Page | Investigate deepagents event stream |
| **Unauthorized delegation blocked** | Any single block event | Warn | Log for security review; alert if > 3 in 1 hour |

### 7.4 Smart Sampling Strategy

Production sampling prioritizes high-risk interactions:

1. **Failed delegations** — 100% logged; 20% sampled for human review (high value signal)
2. **High-artifact-count successes** — `artifacts.length > 5`: 10% sampled (higher downstream risk)
3. **First-time agent slug** — A `task(name)` with a slug never seen before in this session: flag for review
4. **Long task packages** — `package_size > 8000`: 15% sampled (context quality risk)
5. **Stuck `running` tasks** — Always 100% reviewed (possible timeout enforcement failure)

**Calibration cadence:** Every 2 weeks, re-evaluate LLM judge correlation against senior developer labels for artifact quality dimension. Target >= 0.7 correlation before trusting LLM judge scores for pass/fail decisions.
