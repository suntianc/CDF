    Create `src/renderer/src/lib/commands/parseInputToTokens.ts` exporting a named function:

    ```typescript
    parseInputToTokens(
      text: string,
      registry: ReadonlyArray<SlashCommand>
    ): { token?: { name: string; source: CommandSource }; text: string }
    ```

    Behavior — mirror the `dispatcher.resolve()` matcher exactly so the overlay never disagrees with the dispatcher about whether the leading text is a known command:

    1. If `text` does not start with `'/'`, return `{ text }` (no `token` key).
    2. Otherwise, find the first command `c` in `registry` such that:
       - `text === '/' + c.name`, OR
       - `text === '/' + c.name + ' '`, OR
       - `text.startsWith('/' + c.name + ' ')`.
    3. If no match, return `{ text }` (no `token` key).
    4. If a match is found, set `name = c.name` and `source = c.source`. Compute `trailing = text.slice(('/' + c.name).length)` (this preserves the leading space after the command name when present, e.g., `'/goal fix login'` becomes `' fix login'`; e.g., `'/goal'` becomes `''`).
    5. Return `{ token: { name, source }, text: trailing }`.

    Important: the `text` field of the returned object is the raw tail (NOT trimmed). The ChatArea overlay (Plan 03) will use this raw tail to render the user-typed args after the token pill. The 6th acceptance criterion below verifies that `'/goal '` (command + trailing space, no args) returns `text: ''` after the slice strips the leading `'/' + 'goal' + ' '`.

    Do NOT re-parse the `text` tail for additional tokens — per D-03 and SPEC R6, "only one token per input" is a hard constraint. Multiple slashes later in the string are treated as plain text.

    Do NOT import the dispatcher. Mirror the matcher logic in 4 lines of `if` conditions so the helper has zero coupling to the dispatch path.

    Create `src/renderer/src/lib/commands/parseInputToTokens.test.ts` with these 7 cases (each is a separate `it` block):

    1. **Leading known command with args** — `parseInputToTokens('/goal fix login', [goalCmd])` returns `{ token: { name: 'goal', source: 'system' }, text: ' fix login' }` (note the leading space in the tail is preserved, so the rendered overlay can place a non-breaking gap between the pill and the args).
    2. **Leading known command with trailing space only** — `parseInputToTokens('/goal ', [goalCmd])` returns `{ token: { name: 'goal', source: 'system' }, text: '' }`.
    3. **Leading known command, no args, no trailing space** — `parseInputToTokens('/goal', [goalCmd])` returns `{ token: { name: 'goal', source: 'system' }, text: '' }`.
    4. **Unknown command at start** — `parseInputToTokens('/unknown abc', [goalCmd])` returns `{ text: '/unknown abc' }` (no `token` key).
    5. **Non-slash text** — `parseInputToTokens('hello world', [goalCmd])` returns `{ text: 'hello world' }`.
    6. **Empty string** — `parseInputToTokens('', [goalCmd])` returns `{ text: '' }`.
    7. **Multiple slashes (only first parses)** — `parseInputToTokens('/goal /foo bar', [goalCmd])` returns `{ token: { name: 'goal', source: 'system' }, text: ' /foo bar' }` — the second `/foo` is treated as plain text (per D-03: "only first parses").

    Use the same `goalCmd` constant shape that `dispatcher.test.ts` uses (`name: 'goal'`, `source: 'system'`, etc.) so the two test files can be reasoned about side-by-side.
