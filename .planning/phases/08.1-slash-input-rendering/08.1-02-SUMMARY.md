## Decisions Made

- **Named export only (no default).** `export function SlashToken(...)` ‚Äî matches the SlashCommandPopup pattern (forwardRef + named export). The test file imports it as `import { SlashToken } from './SlashToken'`.
- **Icon resolved via the iconMap abstraction.** `const Icon = getSlashTokenIcon(source); <Icon ... />` ‚Äî the component never imports lucide-react directly. This makes the abstraction boundary clean: Plan 03's overlay will never need to know which icon belongs to which source, and iconMap can be swapped (e.g., to a custom icon set) without touching SlashToken.
- **click-no-op verified via createEvent.** The `addEventListener`-based capture pattern fires in the target phase BEFORE React 17+ document-level delegation runs the synthetic handler, so `defaultPrevented` would be read as `false` even when the production handler IS calling preventDefault. Using `createEvent.mouseDown` to obtain the native event, dispatching via `fireEvent`, then reading `event.defaultPrevented` directly on the same event object works because React's synthetic preventDefault updates the native event's `defaultPrevented` synchronously during dispatch.
- **No trailing space or `&nbsp;` inside the span.** The chat text node AFTER the token (rendered in Plan 03) is responsible for the gap. The pill is a static, atomic visual unit.
- **No focus ring, hover state, or animation.** Per SPEC Out-of-Scope ("Token animation / transition") and the neutral-surface intent. A simple static pill is the target.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed click-no-op test pattern to use createEvent (TDD refinement)**
- **Found during:** Task 1 (TDD GREEN verification ‚Äî 8/9 tests passed, 1 click-no-op test failed)
- **Issue:** The original test pattern wrapped `fireEvent.mouseDown` in a `token.addEventListener('mousedown', listener)` capture to read `defaultPrevented` after dispatch. The listener fires in the target phase (capture=false, on the token), but React 17+ event delegation runs the synthetic handler at the DOCUMENT level during the BUBBLE phase ‚Äî so the listener reads `defaultPrevented=false` BEFORE React's handler has a chance to set it to `true`. The handler IS calling preventDefault correctly; the test pattern was just looking at the wrong moment.
- **Fix:** Use `createEvent.mouseDown(token)` to obtain the native MouseEvent, dispatch it via `fireEvent(token, event)`, then read `event.defaultPrevented` directly on the same event object. React's synthetic preventDefault updates the native event's `defaultPrevented` synchronously during dispatch, so the read-after-fireEvent pattern is now reliable.
- **Files modified:** `src/renderer/src/components/SlashCommand/SlashToken.test.tsx`
- **Verification:** All 9 tests pass after the fix; full 47-test phase suite passes.
- **Committed in:** `d8d29e8`

**2. [Rule 1 - Bug] Widened mock-icon prop type for tsc strict mode (TDD refinement)**
- **Found during:** Final tsc strict-mode check
- **Issue:** The lucide-react mock used a narrow prop type (`{ className?: string; 'aria-hidden'?: boolean | string }`) which TypeScript rejected when forwarded into a real `<svg>` element ‚Äî `svg` requires the full `SVGProps<SVGSVGElement>` surface. The narrow type was acceptable at runtime but failed the strict JSX type check.
- **Fix:** Loosen the incoming type to allow extra unknown props and forward them as a single object spread, preserving the `className`/`aria-hidden` reading but satisfying the strict JSX type check. Runtime behavior unchanged: 9/9 tests still pass.
- **Files modified:** `src/renderer/src/components/SlashCommand/SlashToken.test.tsx`
- **Committed in:** `c29fa41`
